# v0.6.6 Milestone

**Target**: TBD (after v0.6.5 ships)
**Bar**: "I can open the inspector, tell the archivist 'I drive a Subaru Outback, Carlie is six not seven, and Jordana Osborne is the same person as Jordana,' and the world state updates accordingly — without those words ever appearing in the narration chat."
**Budget**: ~8 hours — v7 migration + one new agent path + one new route + one new inspector tab + narrator-prompt change.

v0.6.5 shipped the mobile UI pass, the tabbed inspector, and the soft-match canonicalization that the original v0.6.6 doc was meant to do (commit `e0ae15a`, bundled into the v0.6.5 release). With canonicalization handled, v0.6.6 turns to the next gap: the world state is only writable by the archivist as it extracts from narrator turns. The player has no way to assert canon directly — to add a relationship the narrator hasn't introduced, to correct a detail the narrator got wrong, or to tell the archivist that two entities are the same. v0.6.6 adds exactly one new writer of world state — a conversational archivist channel that lives inside the inspector and stays out of the narration.

## Why

1. **The player has no path to assert canon.** "I have a sister named Maeve in Boston" is true at world-creation time but unaddable mid-play. The narrator never invents it; the archivist only records what the narrator wrote. There is no field, no UI, no API.
2. **Archivist mistakes are read-only today.** When the narrator writes "you stepped out of the Suburban" and the archivist files "drives a Suburban" into `memorable_facts`, the only recourse is to play another turn and hope the narrator self-corrects. It won't — the prior fact is in the next prompt.
3. **Canonicalization handles overlapping names but not arbitrary aliases.** The soft-match from v0.6.5 merges "Armitage" with "Professor Armitage" because the tokens overlap. It does nothing for "Bob and Robert are the same person" — no token overlap, no merge. Without a way for the player to tell the archivist explicitly, those rows stay split forever.
4. **Inline field-edit UIs would create a second writer of world state.** Every CRUD surface duplicates the validation, canonicalization, and provenance rules the archivist already implements. A conversational channel reuses the existing patch pipeline and keeps a single writer.

## Scope

One archivist path, one route, one inspector tab, one schema migration. No inline editing, no direct DB writes from the UI, no preview/confirm step — the archivist applies and reports.

## Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Edit shape | **Conversational archivist channel only — no inline field edits, no direct DB writes from UI** | One writer of world state (the archivist) means one validation/canonicalization/provenance code path. Inline editing would duplicate all of that and force a UI for every editable field. The conversational shape also dissolves the schema-is-finite problem: "Carlie is 6" doesn't need an `age` column to land. |
| Where the channel lives | **New "Archivist" tab inside the inspector** | Per explicit requirement: corrections must not appear in the main narration chat. The inspector already has the Now/Story/Wiki tab pattern; adding a fourth is the smallest change that respects the separation. |
| New columns | **`characters.player_notes`, `places.player_notes`** | Provenance: archivist-from-narrator and archivist-from-player must be distinguishable. `player_notes` is the home for player-asserted canon; the archivist's normal extraction path is forbidden from writing it. Without this, the next narrator turn's facts would overwrite player corrections silently. |
| Scrollback storage | **`world_corrections` table with `applied_patch` JSON** | The inspector tab needs to render past exchanges across reloads, and a record of what was applied makes debugging archivist-misroutes possible. JSON blob is sufficient for v0.6.6; a structured audit shape is v0.7+ if needed. |
| Patch shape | **Reuse existing ArchivistPatch schema + add optional `aliases: string[]` on CharacterPatch** | Reuses `applyArchivistPatch` and all its merge/canonicalization logic. The `aliases` field handles the only case the existing soft-match misses: explicit player-declared merges of non-overlapping names ("Bob = Robert"). |
| Reply style | **Executing, not negotiating** ("Updated Carlie's age to 6 and merged Jordana → Jordana Osborne") | Simpler prompt, simpler UI, no multi-turn correction state. If the archivist mis-routes a fact, the reply makes it visible and the player corrects again. Negotiating ("did you mean X?") is v0.7 if executing turns out to misfire often. |
| Apply timing | **Submit applies immediately; no preview/confirm step** | Matches executing-mode. Adding a preview means a second prompt path, a diff renderer, and a confirm UI — three things to build for the rare case where the player wants to abort. Cheap to add later if real. |
| Provenance tag on memorable_facts written via correction | **`[t:N][edit]` where N is the latest assistant turn id** | Keeps existing `[t:N]` rendering working; the `[edit]` suffix lets the inspector mark player-sourced lines distinctly. |
| Narrator awareness of player_notes | **PLAYER CANON block in narrator prompt + soft retcon rule** | Without this, "I drive a Subaru" is recorded but never surfaces in prose. Soft retcon (vs. hard retcon or rewrite) keeps prior turns' append-only invariant — the book has a small scar where Suburban was written, and the narrator moves forward with Subaru gracefully. |
| Archivist write-rules for player_notes | **Correction mode may write it; narrator-extraction mode may not** | Otherwise the archivist will re-overwrite player canon from narrator slips. Enforced in the prompt and, where cheap, in the patch-apply path. |
| Model | **Haiku, same as the existing archivist** | Correction extraction is the same shape of task; cost stays trivial. |
| Dossier corrections (threads/clues/objectives/resources) via chat | **Best-effort; not in exit criteria** | The correction prompt won't actively forbid them and the patch path already supports them, but the prompt is tuned for character/place facts and merges. Promoting dossier corrections to first-class is v0.6.7. |

## Explicit cuts

- **Inline field editing in the inspector** — explicitly out. If the conversational channel proves clumsy for surgical edits, revisit in v0.6.7.
- **Patch preview / confirm step** — v0.7+ if executing-mode misfires often.
- **Archivist negotiating questions ("did you mean X?")** — same.
- **Edit/delete of individual `memorable_facts` lines via UI** — handled via correction chat ("remove the line that says Suburban"). Dedicated UI deferred.
- **Story dossier corrections as first-class** — best-effort via the same chat; not in exit criteria.
- **Audit beyond the `world_corrections` row** — no per-field change log. The applied-patch JSON is enough to reconstruct what changed.
- **Authorization** — no per-user gating on corrections. Multi-user worlds are v0.7+; until then any session that can open the inspector can also correct.
- **Undo** — no undo button. If a correction is wrong, correct again. Considered for v0.7.

## Accepted tradeoffs

- **LLM nondeterminism on writes.** "Make Carlie nicer" is ambiguous (attitude? description? both?). Mitigation: the reply tells you what changed, so misfires are visible immediately and recoverable by a second correction. Acceptable because the alternative (negotiating prompts or preview UI) is much more code for a case that may be rare.
- **Prior prose stays wrong forever.** Turns are append-only — once turn 17 says "Suburban", it always will. The Subaru fix only affects turn 18 onward. The narrator can be coached to retcon gracefully ("you climb back into the Subaru — funny, you'd been calling it the wrong make all morning") but the book has a small scar. Matches the spirit of [[feedback_narration_is_canonical]]: the narration is the book, and the book is append-only. Anything else (rewriting old turns, generating retcon turns) is v0.8 territory.
- **One token of latency per correction.** Each Send is a Haiku call. Cheap, but real. No streaming for v0.6.6 — corrections are short enough to wait for.
- **Vertical real-estate competition on mobile.** A fourth tab in the bottom sheet adds another row. Composer is intentionally compact (single textarea, no toolbar).
- **The archivist could route a fact wrong** (e.g., player says "I drive a Subaru" and it lands in `description` instead of `player_notes`). The prompt biases toward `player_notes` for player-asserted canon, but it's a prompt rule, not a hard guarantee. Visible in the reply; correctable by re-correcting. If it becomes a pattern, tighten the prompt or add deterministic routing in v0.6.7.

## Exit criteria

1. `npm run lint` and `npm run type-check` pass.
2. `npm test` passes; new `tests/archivist-correction.test.ts` covers: (a) "I drive a Subaru" → `player_notes` written, no narrator turn involved; (b) explicit alias merge ("Bob and Robert are the same") collapses to one `characters` row; (c) factual correction ("Carlie is 6 not 7") updates the existing character without inventing a new row; (d) `world_corrections` row is written for each call with reply + applied_patch.
3. `package.json` reads `0.6.6` on the release branch (bumped on the branch itself per [[feedback_release_version_bump]]).
4. v13 migration runs clean on a fresh world and on a copy of prod (PRAGMA user_version = 13 after run; no foreign_key_check violations).
5. Manual (fresh world): open inspector → Archivist tab → type "I drive a Subaru Outback" → Send → reply confirms the change → switch to Wiki tab → player character's `player_notes` shows the line → send a turn in the narration chat → narrator references the Subaru, not whatever it would have invented.
6. Manual (correction over a narrator mistake): play far enough that the narrator writes a detail you want to fix → open Archivist tab → describe the correction → reply confirms → next narrator turn reflects the corrected canon, even if it requires a soft retcon.
7. Manual (merge): introduce two characters whose names the canonicalization heuristic won't collapse (e.g. "Bob" and "Robert") → tell archivist "Bob and Robert are the same person" → Wiki shows one row.
8. Manual (separation): scrolling the main narration chat shows zero trace of any correction text or archivist reply.
9. Manual (persistence): scrollback in the Archivist tab survives a page reload.
10. Deploy → Railway smoke: one correction round-trip on prod, no 5xx; v13 migration ran clean.
11. Tag `v0.6.6` on `main`.

When all hold, v0.6.6 ships. Next: v0.6.7 picks up whichever of {inline surgical edits, preview/confirm, archivist negotiation, dossier-correction polish} the v0.6.6 playtest shows is most needed.

## Open questions

- **Should the Archivist tab show a "what state changed" diff** alongside the reply (collapsible), or is the natural-language reply enough? Default: reply only for v0.6.6; add a "show details" toggle in v0.6.7 if the reply turns out to under-specify.
- **Should the correction prompt see the recent narrator turns** (e.g. last 3) to ground "the Suburban from the last turn"? Cheap to add and probably necessary for context-dependent corrections. Likely yes — confirm during implementation.
- **What's the upper bound on `player_text` length?** Suggest 2000 chars to start; the correction prompt isn't meant for long-form lore drops (use world-creation for that). Adjustable.
