# v0.5.2 Patch

**Target**: 2026-05-25 (shipping on this date)
**Bar**: "Every shipped version has a milestone doc; the README explains where to look; memory matches reality."
**Budget**: ~1 hour. Pure docs + git hygiene. No code changes.

v0.5.0 shipped without a milestone doc, and the loose pile of design docs in `docs/` had no index. While planning v0.6.0 we discovered that the original v0.5 typed-state work (commit `95dbe85`) had been orphaned — present in the reflog, present nowhere else. v0.5.2 closes those gaps before v0.6.0 opens.

## Why

1. **v0.5.0 was undocumented.** A reader landing on the repo today couldn't tell what shipped in v0.5.0 without diffing tags. The auto-memory entry for v0.5 was confidently wrong, describing typed-state work as shipped.
2. **The typed-state commit was at GC risk.** It existed only in the reflog and would have been garbage-collected within the default 90-day window.
3. **The docs directory had no entry point.** A new contributor (or future me, six months in) would have to read all 17 files to know which were design specs and which were milestones.
4. **No milestone template.** Each existing milestone doc was written from scratch, with the shape drifting between versions. Future docs needed a copy-and-fill template.

## Scope

Six new files, two file edits, one branch rescue, two annotated git tags. No production code touched.

## Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Document v0.5.0 honestly vs. backdate the typed-state plan as "shipped" | **Honest backfill** | Future debugging depends on doc-vs-reality alignment. Backdating creates a worse version of the problem we just discovered. |
| What to do with the orphaned typed-state commit | **Save to local branch, push to origin** | Zero-risk preservation. Decoupled from the question of when/whether to merge it. |
| Whether to rebase `save/v0.5-typed-state` onto current main as part of v0.5.2 | **No — leave it frozen** | Conflating rescue with rebase invites scope creep. The rebase is a v0.6.0 sprint task that needs smoke-testing and a DB snapshot. |
| Numbering for the v0.4.2 backfill | **18 (next sequential), not "14.5" or inserted near v0.4.1's doc** | Filenames sort by write-order, not version-order. Renumbering existing files would invalidate every link in the wild. |
| Where v0.5.0 backfill lives | **Fill the existing `15-` gap** | `15-` was already an implied placeholder; the originally-planned v0.5 design doc was numbered 15 in the orphaned commit. Reusing the slot keeps version-order and write-order aligned for this one case. |
| Doc template format | **One file, embedded inline examples, no separate examples directory** | The template's shape is the point. A reader copies the file once and edits it. |
| Memory hygiene scope | **Rewrite affected entries entirely, not patch in place** | The old `v0_5_milestone.md` was 38 lines of confidently-wrong claims. Patching would leave landmines. |
| Whether to remove `APP_PASSWORD` from Railway env in this patch | **Yes, alongside the tag pushes** | Touches prod env (small blast radius — the variable is already unused). Bundled with the deploy-time work, not the local work. |

## Explicit cuts

- **No retroactive tag rewrite for v0.5.0.** The thin annotation stays; this doc supersedes it.
- **No GitHub Release notes backfill.** Worth doing eventually but not in scope here.
- **The `save/v0.5-typed-state` branch is not rebased.** That's the entire point of v0.6.0.
- **No design-doc rewrites.** `01`–`09` continue to describe the long-term target; bringing them in line with current implementation is its own pass.

## Accepted tradeoffs

- **The README claims one doc per version, and v0.5.2 itself is the first patch under that rule.** If we backslide (skip a doc for a tiny patch later), the convention erodes. The cost of the rule is consistent; the cost of breaking it compounds.
- **The typed-state work continues to drift relative to main.** Every additional v0.5.x patch makes the eventual v0.6.0 rebase slightly harder. Accepted because parking-with-drift is still cheaper than rebasing-on-every-patch.
- **The memory rewrite loses the original wording.** Anyone digging through `~/.claude` history could find the prior version; not a concern in practice.

## Exit criteria

1. `docs/_template-milestone.md`, `docs/README.md`, `docs/15-v0.5.0-milestone.md`, `docs/18-v0.4.2-patch.md`, and this file (`docs/19-v0.5.2-patch.md`) all exist on `main`.
2. `package.json` reports `0.5.2`.
3. Annotated `v0.5.1` tag exists at `6b442f7` on `main`, pushed to origin.
4. Annotated `v0.5.2` tag exists at the merge of this work, pushed to origin.
5. `save/v0.5-typed-state` branch exists on `origin`, pointing at `95dbe85`.
6. `APP_PASSWORD` is removed from the Railway production environment.
7. Memory entries match reality: `v0_5_milestone.md` describes auth-only, `v0_5_1_audio_chunking.md` notes the missing tag + iOS smoke test, `save_v0_5_typed_state_branch.md` exists.

When all seven hold, v0.5.2 ships. Next milestone: v0.6.0 (rebase `save/v0.5-typed-state` onto main; typed world-state + archivist).
