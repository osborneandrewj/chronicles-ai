# v0.6.2 Milestone

**Target**: TBD (after v0.6.1 ships)
**Bar**: "The play view is comfortable on a phone — composer, inspector drawer, and per-turn footer all work one-thumbed; narrator audio plays on iOS Safari. Replaying an old narrator turn updates that turn's cost footer, not the latest one."
**Budget**: TBD — size after the mobile design call lands.

v0.6.1 ([`docs/21-v0.6.1-milestone.md`](21-v0.6.1-milestone.md)) tightens the narrator prose, gives NPCs immediate goals, and clears the post-v0.6.0 audit findings. v0.6.2 picks up the two pieces that were intentionally pulled out of v0.6.1 plus an iOS audio failure surfaced during the v0.6.1 Railway smoke test: the mobile + desktop UI pass, the iOS audio bug, and a per-turn cost-footer bug. No new tables, no new agents, no new env vars.

## Why

Three motivating problems:

1. **The play surface assumes a desktop browser.** Andrew + Joe play on phones; the inspector drawer, chat composer, and per-turn footer all need a mobile pass. Desktop also has small comfort wins worth bundling.
2. **Narrator audio is silent on iOS Safari.** Smoke-tested 2026-05-25 on the v0.6.1 Railway deploy: tapping the audio toggle to "Audio on", sending a turn, and tapping any prior turn's Replay button all produce **total silence — no `speaking` indicator, no sound**. Desktop Safari works fine on the same deploy. Codex confirmed via Railway HTTP logs that `/api/tts` returns 200s on mobile (audio bytes are arriving), so the failure is downstream of the network. Root cause: the iOS `AudioContext` is still in `suspended` state when `source.start()` fires asynchronously after the TTS fetch resolves — iOS Safari silently drops playback in that case. The v0.5.1 Web Audio rewrite (commit `749e3f8`) was meant to fix exactly this, but the memory file admits it was never actually verified on a real iOS device. This is that fix landing on a phone for the first time, and it's not enough.
3. **Replaying an old narrator turn doesn't update that turn's cost footer.** v0.6.0 wired `addTtsChars(turnId, chars)` to make replay-of-old-turn additive on the right row, but the smoke test showed the per-turn footer next to the replayed turn does not move. The audio plays (when it plays — see #2), the cost stays wrong. Low priority; visible to anyone using the replay control. See [[v0_6_0_replay_cost_bug]].

## Open questions

- **Mobile target devices:** confirm iPhone SE as the floor. Landscape — must-work or nice-to-have?
- **Inspector at narrow widths:** full-screen sheet, bottom drawer, or hidden behind a route push? (Leaning sheet.)
- **Composer behaviour with software keyboard:** does the inspector close when the keyboard opens, or stay mounted underneath?
- **Per-turn footer at <360px:** wrap to two lines, hide the per-turn cost behind a tap, or shrink the text? Each has a cost.
- **Replay-cost root cause:** which of the three hypotheses in [[v0_6_0_replay_cost_bug]] is it? Diagnose before committing to a fix shape.
- **iOS audio — does fix #2 alone resolve it, or do we need #1+#3 too?** Plan is to ship all three since they're all cheap; if the mobile diagnostic panel shows fix #2 is sufficient, fix #1 (don't-close-ctx) is still worth keeping on its own merits (closing then reopening a ctx on hook remount is a latent bug). Fix #3 (longer prime buffer) is harmless either way.
- **Does the audio bug have a desktop manifestation we missed?** Desktop currently works, but the same suspended-ctx race could fire after backgrounding (open the tab → audio works → switch to another app → return → audio suspended). Worth a manual check during v0.6.2 smoke.

## Explicit cuts

- **No new agents, no new tables, no new env vars.** v0.7 territory.
- **No Android-specific work.** Until there's an Android user.
- **No PWA install / offline.** Out of scope; mobile-web is fine for v0.6.2.
- **No keyboard shortcuts pass for desktop.** Separate concern; revisit when a power-user workflow demands it.
- **No virtualised chat history.** v0.6.1 ships the pagination cutoff; virtualisation stays parked until the 60-turn initial load feels slow.

## Accepted tradeoffs

- **Hand-rolled mobile sheet means more JSX in `WorldInspector.tsx`.** Tolerable for one surface; would push us to a library if a second arrives.
- **No automated viewport tests.** Mobile QA is manual on real devices. CI snapshot tests on a mobile viewport were considered and rejected — they catch layout regressions but not touch ergonomics, and the maintenance cost is high for a two-person team.
- **Landscape iPhone is "supported but not optimised".** A landscape user on a Pro Max will see desktop-shaped chrome on a short viewport. Accepted because the primary use case is portrait.
- **Replay-cost fix only addresses the per-turn footer.** The session total in the header is already correct via `addTtsChars`; if it isn't, that's a separate bug to chase.

## Exit criteria

*(Mobile criteria sharpen after the design call.)*

1. `npm run lint` and `npm run type-check` pass.
2. `npm test` passes, including the updated `useNarratorAudio` test for the replay path.
3. `package.json` reads `0.6.2` on the release branch (bumped on the branch itself per [[feedback_release_version_bump]]).
4. Manual (replay-cost): replay an old narrator turn (not the latest). Per-turn footer next to that turn grows by the replay's char count; session total in the header grows by the matching cost. Replay the latest assistant turn — its footer also grows, no cross-turn leakage.
5. Manual (iOS audio — fresh session): on iPhone Safari → open a world → ensure header shows "Audio on" → send a turn → narrator response streams in **with audible playback**; speaking indicator (amber dot) fires. Then tap Replay on any prior turn → audible playback. **No silent failures.** This is the smoke test that uncovered the bug; it's the canonical pass condition.
6. Manual (iOS audio — backgrounded session): on iPhone Safari with audio working, switch to another app for 30s, return → next send still produces audible audio (verifies fix #2 handles the suspended-ctx race after backgrounding, not just on first load).
7. Manual (mobile chat): on iPhone (375px) → open a world → compose and send a turn → narrator response renders without horizontal scroll, per-turn footer is legible, composer remains reachable above the software keyboard.
8. Manual (mobile inspector): on iPhone (375px) → toggle inspector → drawer/sheet covers the chat without breaking it underneath → close affordance is thumb-reachable → reopen shows current state.
9. Manual (desktop regression): on a 1440px viewport, all v0.6.1 behaviour still works — inspector drawer, audio toggle, slash command menu, per-turn footer, audio playback.
10. Deploy to Railway → Andrew + Joe each play one turn on phone and one on desktop. No 5xx. Inspector usable on both. Audio audible on both.
11. Tag `v0.6.2` on `main`.

When all hold, v0.6.2 ships. Next milestone candidates: v0.7 (embeddings + `[t:N]` consumer code, per-character voices, Postgres migration, CRUD on inspector entities — see the v0.6.0 "next sprint" note).
