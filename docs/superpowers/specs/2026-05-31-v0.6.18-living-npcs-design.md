# v0.6.18 — Living NPCs: Substrate

**Status:** Approved design (2026-05-31)
**Target version:** v0.6.18
**Follow-on:** v0.6.19 (emergent-behavior layer — see "The v0.6.19 seam")

## Summary

v0.6.18 lays the deterministic substrate for a more alive NPC system: it fixes a
reverie data-loss bug, makes reveries **flare on demand** through deterministic
tag-matching instead of LLM guesswork, and gives NPCs **daily loops** that supply
a baseline rhythm and make off-scene gap-fill a cheap lookup instead of an LLM call.

This is the first of a two-part effort. The emergent-behavior layer
(cornerstone/awakening, off-screen advancement, proactive NPC-initiated contact)
is deferred to v0.6.19 and is designed to read the rails this version lays down.

### Guiding principle

Every new mechanic is **deterministic code** (free, testable, reproducible) except
where authored richness genuinely needs an LLM — and where it does, the LLM call
**piggybacks an existing NPC-agent tick** so steady-state token cost goes *down*,
not up. This keeps us inside the 8K-in / 1K-out narrator budget and improves the
budget story rather than straining it.

### Decisions (locked during brainstorming)

- **Reverie storage:** dedicated structured `npc_reveries` table (not JSON, not text).
- **Loop generation:** authored once by the NPC agent (Haiku) at first opportunity
  after promotion, piggybacking the existing tick — no separate call.
- **Release split:** this version ships the substrate only. Cornerstone/awakening,
  off-screen advancement, and proactive contact are v0.6.19.

### Out of scope (explicitly v0.6.19)

- Cornerstone memory + dissonance/"awakening" track (subtle-coloring ceiling).
- Off-screen advancement for principal NPCs (scoped active director).
- Proactive NPC-initiated contact (turn-driven intrusion).
- Loop **deviation/dissonance tracking**. v0.6.18 stores and *uses* loops; it does
  not yet score deviation from them (that is awakening's raw signal, in v0.6.19).

## Mechanic 1 — Reverie durability (the bug fix)

### Problem

`characters.reveries` is a newline-text column written as an **overwrite** by the NPC
agent (`npc-agent.ts:503`, `:569-570`), where the schema instructs the model to
"include prior reveries you want to keep" (`npc-agent.ts:55-60`). But the agent is
only shown `lastNLines(stripFactProvenance(a.reveries), 4)` (`npc-agent.ts:344`) —
so once an NPC has more than four reverie lines, the older ones are **not in the
prompt** and the model literally cannot re-include them. They are silently
truncated to oblivion on the next write. The most identity-defining field is the
least durable — backwards.

### Fix

Replace the overwrite-with-reinclude column with an **append-only structured table**.
Because rows are inserted and never bulk-overwritten, the bug disappears by
construction: the agent no longer re-emits prior reveries at all.

### Schema (migration 24)

```sql
CREATE TABLE npc_reveries (
  id                   INTEGER PRIMARY KEY AUTOINCREMENT,
  world_id             INTEGER NOT NULL REFERENCES worlds(id) ON DELETE CASCADE,
  character_id         INTEGER NOT NULL REFERENCES characters(id) ON DELETE CASCADE,
  text                 TEXT    NOT NULL,
  match_tags           TEXT    NOT NULL DEFAULT '',  -- normalized, comma-separated
  intensity            REAL    NOT NULL DEFAULT 0.5
                         CHECK (intensity >= 0 AND intensity <= 1),
  is_cornerstone       INTEGER NOT NULL DEFAULT 0,   -- reserved; unused until v0.6.19
  created_turn_id      INTEGER REFERENCES turns(id) ON DELETE SET NULL,
  last_flared_turn_id  INTEGER REFERENCES turns(id) ON DELETE SET NULL,
  created_at           TEXT    NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_npc_reveries_character ON npc_reveries(character_id);
```

- **Backfill:** the migration splits each existing `characters.reveries` newline into
  a row (`text` only; `match_tags` empty; `intensity` 0.5; `created_turn_id` null).
- **Dormant column:** `characters.reveries` is left in place (not dropped) but is no
  longer written or read after this migration. Dropping it is a later cleanup.

### Authoring (NPC agent)

`NpcUpdateSchema` (`npc-agent.ts:17-90`) drops the overwrite field and gains an
append-only one:

```ts
reveries_add: z
  .array(
    z.object({
      text: z.string(),
      match_tags: z.array(z.string()).default([]),
      intensity: z.number().min(0).max(1).optional(),
    }),
  )
  .optional()
  .describe(
    'Add NET-NEW reveries only. Never repeat existing ones — they persist on their own. ' +
      'Tag each with concrete sensory/emotional anchors (smell, phrase, object, place, failure). ' +
      'Add a reverie rarely, only when something charged genuinely lodges.',
  )
```

- The agent is shown the NPC's existing reverie texts **for dedup awareness only**
  (so it doesn't re-add a near-duplicate). It is no longer responsible for
  preserving them.
- **Pruning rule:** cap each NPC at ~6 reveries. On insert past the cap, evict the
  lowest-`intensity`, least-recently-flared row. Dedup new adds by normalized text.

### Persistence

Replace the overwrite `UPDATE characters SET reveries = ?` path (`npc-agent.ts:503`,
`:569-570`) with `INSERT INTO npc_reveries (...)` for each entry in `reveries_add`,
followed by the prune pass. Omitting `reveries_add` is a no-op — it must **never**
delete existing rows (pinned by a regression test).

## Mechanic 2 — Deterministic reverie flare-matching

### Goal

A reverie should *flare* when the current scene "rhymes" with it (the literal
Westworld mechanic the feature is named after). Today the rhyme is detected by
nobody — the agent writes a string, `formatStateBlock` dumps it as ambient text,
and the narrator is loosely told to honor it. We make the flare **deterministic and
measurable**.

### Scene tags

Build a `sceneTags: string[]` for the turn from **structured sources only** (no raw
player-text tokenization, to stay reproducible):

- the active place's `match_tags` (place templates already define these,
  `place-population.ts:119-128`);
- the active dossier threads' `relevance_tags`;
- the occupancy / encounter-hook tags from the `turnOccupancy` snapshot already
  built at `route.ts:226`.

### Flare computation (pure function)

New `computeReverieFlares(reveries, sceneTags, opts)` in a dedicated module:

- For each reverie: `score = |overlap(match_tags, sceneTags)| * intensity`.
- Per NPC, the single highest-scoring reverie with **≥1 tag overlap** flares.
- Global cap: **≤2 flares per turn**, present NPCs preferred, then highest score.
- Stamp `last_flared_turn_id` on the chosen rows.

Pure and deterministic → fully unit-testable; no tokens.

### Surfacing

`formatStateBlock` (`world-state.ts:290-298`) now reads reveries from the table.
Non-flaring reveries render as today (capped ambient list). A flaring reverie
renders distinctly:

```
- ⚡ REVERIE FLARING — must surface this turn as a physical tell / hesitation /
  misread, never as narrated memory-exposition: "<text>"
```

The narrator prompt (`narrator-system.md:54`) gains a matching hard rule: a flaring
reverie **must** manifest as action/subtext this turn and **must not** be narrated
as mechanical memory exposition.

This is the line that converts reveries from hoped-for decoration into a load-bearing,
queryable behavior driver (we can now measure flare frequency and whether prose
honored a flare).

### Character merge

The archivist's character dedup currently merges reverie text via `mergeLineBlocks`
(`archivist.ts:1465`). With rows, merge instead re-points source `npc_reveries` rows
to the target `character_id` and dedups by normalized text.

## Mechanic 3 — NPC daily loops

### Storage

Add `characters.daily_loop` (nullable JSON text; migration 24). A loop is read as a
whole unit per NPC, so a column beats a table here.

```json
{
  "morning": { "activity": "opens the shop",                 "place": "Tin Anchor" },
  "midday":  { "activity": "market run, gossips with Tomas",  "place": "South Market" },
  "evening": { "activity": "tends the wayside shrine",        "place": "Wayside Shrine" },
  "night":   { "activity": "drinks late",                     "place": "Tin Anchor" }
}
```

### World-time bands (new helper)

`world_time` is a **free-text** string (e.g. "Day 3, 3pm"); there is no existing
band parser (place-population only hashes the string into its PRNG seed). Introduce
a deterministic helper:

```
worldTimeBand(world_time: string): 'morning' | 'midday' | 'evening' | 'night'
```

It parses common clock formats and keywords (numeric "3pm"/"15:00", and words like
"dawn"/"morning"/"noon"/"dusk"/"night") with a defined fallback (default `midday`)
when the string is unparseable. Pure and unit-tested. (Aligning place-population to
this helper is a non-goal for v0.6.18 — out of scope to avoid unrelated refactoring.)

### Generation — zero new LLM calls

`daily_loop` is added as an optional field on the **existing NPC-agent output
schema**. The system prompt (`npc-agent-system.md`) instructs: *"If you have no
daily_loop yet, author one now"* (time-banded activities + the place each happens
in). Because `local` NPCs already tick every turn, the first tick after promotion
fills the loop once; later ticks leave it untouched (only an explicit
regeneration directive replaces it). No separate call, no added hot-path latency.

`npc-promotion.ts` is unchanged except to guarantee a newly promoted `local` NPC
will tick; the agent does the authoring.

### Usage — steady-state cost goes down

- **Off-scene gap-fill becomes a lookup.** Today every off-scene `nearby`/`distant`
  NPC needs a Haiku tick purely to emit `activity_append`. With a loop, that line is
  `activityForBand(daily_loop, worldTimeBand(world_time))` — deterministic. NPCs
  whose only update is routine activity can skip the LLM tick entirely, removing
  Haiku calls per turn.
- **Continuity.** The current band's activity is injected into the off-scene NPC
  block in STATE ("by routine, Tomas is at the South Market now"), so re-entry stays
  consistent and the narrator cannot contradict it.
- **Plan grounding.** The loop is shown to the NPC agent so present-NPC plans respect
  the routine.
- **Overrides.** `in_transit_to_place_id` (journey state) suspends the loop; an
  explicit narrated action overrides the band for that turn. v0.6.18 only stores and
  uses loops — it does not score deviation (that is v0.6.19).

## Testing

All the hard logic is pure and deterministic:

- `computeReverieFlares(reveries, sceneTags)` — overlap/intensity scoring, the
  ≥1-overlap threshold, the ≤2-per-turn cap, present-NPC preference, and
  `last_flared_turn_id` stamping.
- `worldTimeBand(world_time)` — numeric and keyword parsing plus the fallback.
- `activityForBand(loop, band)` — band lookup, including a null/partial loop.
- **Append-only regression (the bug):** an NPC-agent tick that omits `reveries_add`
  must never delete existing `npc_reveries` rows.
- **Migration 24:** newline `characters.reveries` splits into N rows; dormant column
  untouched; `daily_loop` column added.
- **Pruning:** a 7th reverie evicts the lowest-intensity / least-recently-flared row.
- **Zod:** `reveries_add` accepted; legacy `reveries: string` rejected.
- **Character merge:** source reverie rows re-point to target and dedup by text.

## Cost / budget

Flare-matching and loop lookup are deterministic (zero tokens). Loop generation
piggybacks an existing tick (no new call). Off-scene gap-fill via lookup *removes*
Haiku calls. Net steady-state token cost should **drop**.

## Migration / deploy / "done"

- One migration (**24**): create `npc_reveries`, add `characters.daily_loop`,
  backfill reveries from the dormant column.
- Version bump on the `v0.6.18` branch per the release rule: `package.json` **and**
  `package-lock.json` (both version fields), single commit; restart the dev server
  (module-level JSON import is not HMR'd); update the milestone exit criteria.
- **Done** = a turn streamed end-to-end in the browser where (a) a tagged reverie
  visibly flares and the prose honors it as subtext, and (b) an off-scene NPC's
  routine activity shows up correctly on re-entry. Inspector updated to read reveries
  from the table (showing intensity / flaring / cornerstone fields).

## The v0.6.19 seam

v0.6.18 lays all the rails the emergent layer will ride; it adds the trains:

- `npc_reveries.is_cornerstone` already exists — cornerstone memory is a flag + a
  selection rule on the same table.
- `characters.daily_loop` + journey state are exactly what off-screen advancement and
  proactive contact will read to decide *what* an NPC is doing and *why it would
  reach out now*.
- `npc_intents` dispositions (`contradicted` / `ignored`, already recorded) plus
  loop-deviation are the raw signal the dissonance/"awakening" metric will
  accumulate (ceiling: subtle coloring — hesitate, question, misremember, act
  slightly against goal; never fully break the authored story without a tunable
  config knob).
- Proactive contact (v0.6.19) is **turn-driven intrusion**: on the player's next
  turn an NPC may reach out (approach / phone / knock / message) when loop + goals
  warrant it, reusing the momentum/narrator path — no async push, no scheduler.
