# v0.6.4 Milestone

**Target**: TBD (after v0.6.3 ships)
**Bar**: "The narrator has mission-shaped memory: active threads, clues, objectives, resources, and hidden pressure are visible in the prompt as playable story pressure instead of scattered entity facts."
**Budget**: TBD — schema + archivist patch + narrator digest + inspector surface.

v0.6.2 tightened narrator prose and v0.6.3 isolated the Safari/WebKit audio fix. v0.6.4 is the next narrative-memory step: keep SQLite as canonical storage, but add a Markdown-shaped dossier layer so the narrator can reason from current assignments, open questions, clues, and clocks instead of reaching for repeated ambient closers.

(Mobile UI + replay-cost-footer work moved to v0.6.5 — see [`docs/plans/milestones/v0.6.5.md`](v0.6.5.md). Character canonicalization moved to v0.6.6 — see [`docs/plans/milestones/v0.6.6.md`](v0.6.6.md).)

## Why

Recent playtests show the current memory is too entity-shaped:

1. **The narrator knows setting facts but not story pressure.** It can see wheat, rain, Vox, a fragment, and a place. It does not reliably see "the active investigation is identifying a fresh Imperial relay fragment and the next leads are Stygies VIII, the spire, and who planted it."
2. **SQLite tracks people and places well, but not playable branches.** Characters, places, observations, and facts are good infrastructure. They do not directly represent open threads, active objectives, clues, contradictions, debts, clocks, or hidden pressure.
3. **A pure Markdown wiki is ergonomic for humans but weak as app state.** Grok's campaign wiki structure is useful because it is story-shaped (`Quests.md`, `Evidence_Log.md`, `Timeline.md`, character sheets), not because Markdown should replace the database.
4. **The narrator needs licensed inference.** For investigation/tool/NPC-answer turns, the narrator should produce information, partial information, or a specific obstacle. A dossier can tell it which discoveries are available and which mysteries should remain occluded.

## Proposed schema

Keep this deliberately boring and queryable:

```
story_threads
  id, world_id, title, kind(quest|mystery|threat|relationship|background),
  status(active|resolved|failed|dormant), summary, stakes, rewards,
  consequences, hidden, source_turn_id, resolved_turn_id, created_at, updated_at

story_clues
  id, world_id, thread_id nullable, title, detail, implication,
  status(open|interpreted|spent|false_lead), source_turn_id, created_at, updated_at

story_objectives
  id, world_id, thread_id nullable, title, status(active|blocked|completed|failed),
  detail, blocker, source_turn_id, completed_turn_id, created_at, updated_at

story_resources
  id, world_id, owner_character_id nullable, name, kind, status,
  detail, source_turn_id, created_at, updated_at

timeline_events
  id, world_id, turn_id, thread_id nullable, world_time nullable,
  title, summary, importance(1..5), created_at
```

`kind = quest` gives the narrator a Quests.md-shaped mission container without adding a parallel table. `hidden` on story_threads is narrator-visible but should not be dumped as player-facing exposition. It is for pressure: who acts, what approaches, what remains occluded.

## Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Canonical source | **SQLite** | The app already relies on typed queries, migrations, provenance, and inspector state. Markdown as canonical runtime state would make consistency and deploy behavior harder. |
| Prompt shape | **Markdown-like dossier digest** | The Grok wiki lesson is structure: active quests, evidence, timeline, character sheets. Render that shape into the narrator prompt without moving storage to files. |
| Human-editable Markdown | **Deferred** | Useful later. v0.6.4 should first prove the story model. Export/import can ship once the schema stabilizes. |
| Dossier updater | **Archivist patch extension** | The archivist already sees the latest player+narrator turns and writes state. Adding story-memory fields there keeps one post-turn state pipeline. |
| Hidden information | **Store hidden pressure separately from clues** | The narrator can use hidden pressure to make the world act without blurting secrets. Player-visible clues remain separate. |
| Inspector editing | **Read-only** | Editing objectives/clues by hand is attractive, but not needed to test whether the narrator improves. Manual edit UI is v0.7 territory. |
| Markdown files | **Optional export later** | A future `campaign_wiki/` export can render `Schema.md`, `Timeline.md`, `Characters/*.md`, and `Evidence_Log.md` from SQLite. Not runtime in v0.6.4. |

## Open questions

- **How much hidden pressure should the narrator see?** Enough to make NPCs/world events move; not enough to spoil every mystery in prose.
- **Should `resources` duplicate character memorable facts?** Lean yes for tools/companions/status that affect play, with the archivist avoiding duplicate prose facts when possible.
- **Do objectives belong under threads only, or can they stand alone?** Schema allows both. Some objectives are scene chores; some are plotline milestones.
- **How aggressively should timeline_events record routine turns?** Lean importance 3+ only; clue/objective/resource changes create timeline entries, atmospheric beats do not.

## Explicit cuts

- **No Markdown wiki as source of truth.** Export/import later.
- **No embeddings or semantic retrieval.** This is structured current story memory, not vector memory.
- **No manual dossier editor.** Inspector is read-only for v0.6.4.
- **No retroactive extraction of every old turn.** Seed from current state and extract forward. Backfill can be a one-shot utility later.
- **No separate plot-manager agent.** Archivist extension first; add a dedicated agent only if patches get too noisy.

## Accepted tradeoffs

- **Archivist prompt grows.** Acceptable because the current failure is missing story pressure. Keep per-field limits tight and omit no-op arrays.
- **Some clues may be too vague or duplicated early.** Tests and inspector visibility should make this tunable. Bias toward fewer, more playable entries.
- **Hidden pressure is a trust fall.** The narrator must use it to create action, not expose it. Prompt rules carry that line; playtests decide if it needs code gating.
- **Read-only inspector means corrections still require DB/manual tooling.** Fine for v0.6.4. We need the model first, editor later.

## Exit criteria

1. `npm run lint` and `npm run type-check` pass.
2. `npm test` passes, including dossier schema/apply/digest tests.
3. `package.json` reads `0.6.4` on the release branch.
4. Manual: in a fresh investigative world, finding an object creates an active thread and at least one clue; inspector shows them.
5. Manual: asking a tool/companion to analyze the clue produces a concrete result, partial result, or obstacle; no repeated processing-only beats.
6. Manual: after two quiet turns, the narrator uses active objectives/clues to open a new playable branch instead of repeating ambient setting motion.
7. Manual: resolving an objective marks it completed/resolved and the narrator stops presenting it as active pressure.
8. Manual: hidden pressure can trigger an event without revealing the hidden field as exposition.
9. Deploy to Railway -> one turn end-to-end, no 5xx; inspector Dossier section renders.
10. Tag `v0.6.4` on `main`.

When all hold, v0.6.4 ships. Next: v0.6.5 (mobile UI + replay-cost-footer — see [`docs/plans/milestones/v0.6.5.md`](v0.6.5.md)).
