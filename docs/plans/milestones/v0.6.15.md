# v0.6.15 — Character Dedup Prevention (defense-in-depth, flag-only)

**Target**: 2026-05-29.
**Bar**: "The narrator never quietly forks a person into two rows — when a descriptor like 'the courier' turns out to be Elena Vasquez, the system collapses them to one character, and any likely phantom duplicates are surfaced in the inspector before they pollute the wiki."
**Budget**: ~6-8 hours of focused work.

Patch on top of `v0.6.13`. v0.6.13 delivered archivist name-reveal prompt fixes; this patch adds a defense-in-depth stack that keeps the character roster clean when names are revealed mid-arc. Three layers, all flag-only (no auto-merge, no migration, no new agent):

1. **L1 — Descriptor flagging.** The archivist context now marks placeholder-style descriptor entries so the LLM can recognize a reveals_name_of situation and route it correctly, rather than silently creating a second row.
2. **L2 — Deterministic duplicate detector.** A pure-code `findLikelyDuplicateCharacters` function surfaces character pairs that look like aliases by name-key overlap, near-identical names, or shared-descriptor rules. Surfaced in three places: the inspector's Wiki→Characters tab, the chat-route log after each archivist apply, and `merge-characters.mjs --detect`.
3. **L3 — `reveals_name_of` rename channel.** The archivist can now emit a `reveals_name_of` patch field; the chat route intercepts it and routes the rename through the existing `runAliasMerges` pipeline, persisting the descriptor as an alias on the merged row.

No new tables, no schema migration. No auto-merge (human confirms via `merge-characters.mjs`). Built on top of the v0.6.13 archivist name-reveal prompt fix.

## Why

1. **Descriptors silently fork into two rows.** When the narrator reveals that "the courier" is Elena Vasquez, the archivist would create a new Elena row instead of merging — leaving both alive, causing the wiki to show duplicates and the narrator to see both in context.
2. **There was no code-side signal for likely duplicates.** Without a detector, duplicates only surfaced when a human noticed the wiki looked wrong. By then the drift could span many turns.
3. **The rename path bypassed alias persistence.** Even when the archivist correctly flagged a rename, the alias (the descriptor form) was dropped. Later turns could re-introduce the descriptor as a fresh anonymous character.

## Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Flag-only, no auto-merge | Human confirms via `merge-characters.mjs` | Auto-merge on partially-matched names risks collapsing distinct characters (e.g. two couriers). Safe default: surface, don't act. |
| Detector rules | name-key overlap, near-identical names, shared-descriptor label | Covers the main failure modes deterministically, without an LLM call in the turn path. |
| reveals_name_of routing | Intercept in chat route, delegate to runAliasMerges | Reuses the existing merge pipeline; the alias row is handled by code that already manages it. |
| Descriptor persistence | Alias on the merged row | Keeps old references valid; later narration referring to "the courier" still resolves to Elena. |
| No migration | Reuse existing schema | The alias column already exists; reveals_name_of is a patch-time signal, not a stored column. |
| L2 surfaces in three places | Inspector + chat log + CLI detect | Different audiences: players see the inspector, devs see the log, ops use the script. One detector function, three call sites. |

## Architecture sketch

```
archivist emits reveals_name_of: "the courier"
        │
        ▼
chat route intercepts patch
        │
        ▼
runAliasMerges(worldId, targetId, descriptorAlias)
        │
        ├─▶ descriptor row deleted / merged into target row
        └─▶ alias persisted on target row

findLikelyDuplicateCharacters(worldId, db)
        │
        ├─▶ name-key overlap rule
        ├─▶ near-identical name rule
        └─▶ shared-descriptor rule
              │
              ▼
        DuplicatePair[] → inspector / chat log / --detect
```

## Explicit cuts

- **Auto-merge** — deferred; requires higher confidence rules and a review UX.
- **Semantic / embedding-based dedup** — deferred to Phase 2 (Voyage AI); deterministic rules cover the main cases now.
- **Story Conductor / escalation** — still a stub; deferred to `v0.6.14`.
- **Drift detection / off-screen advancement** — deferred to `v0.6.14`.
- **Background cron dedup sweep** — deferred; the detector runs in-process on demand for now.

## Accepted tradeoffs

- **Detector has false negatives.** Two characters with completely different name-keys and no shared descriptor won't be caught. The narrator prompt and archivist instructions reduce the upstream failure rate; the detector catches the residual.
- **Detector has false positives.** Common descriptors ("the man", "the woman") could match unrelated characters. The inspector shows pairs for human review — no automatic action is taken.
- **L3 fires only when archivist emits the field.** If the archivist misses a reveal, the merge doesn't happen. L2 catches the residual drift at the next inspector check.
- **No migration means old phantom rows persist.** Existing duplicates must be resolved manually via `merge-characters.mjs`. The detector makes finding them cheap.

4. Archivist `reveals_name_of` collapses a descriptor→proper-name reveal to one character row, persisting the descriptor as an alias on the merged row (covered by automated tests).
5. A seeded duplicate character pair is surfaced in the Wiki→Characters inspector tab, appears in the chat-route log after archivist apply, and is listed by `node scripts/merge-characters.mjs --detect <worldId>`.
6. Deploy to Railway, smoke one turn, confirm no 5xx or merge-path errors.
