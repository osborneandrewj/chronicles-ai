# v0.4 Milestone

**Target**: 2026-07-06 (~2 weeks from 0.3.0)
**Bar**: "I play a turn, the narrator's prose streams in and is spoken aloud sentence-by-sentence in a voice I picked, and I can mute it."
**Budget**: ~6–8 hours of focused work.

0.1 proved the streaming loop. 0.2 closed per-turn quality gaps. 0.3 opened the schema with `worlds`. 0.4 is a deliberate one-feature sprint: **text-to-speech narration via the xAI Grok TTS API**, played back sentence-by-sentence as the narrator streams. No schema changes, no characters table, no Postgres port. The point is to test whether a spoken narrator changes the feel of play before investing further.

The planned v0.4 `characters` work moves to v0.5. One structural change at a time.

## Why xAI TTS

Surveyed in May 2026:

- **xAI Grok TTS** — `https://api.x.ai/v1/tts`, REST server-streamed audio + WebSocket bidirectional. 5 voices (Eve, Ara, Leo, Rex, Sal). Inline expressive tags (`[laugh]`, `[sigh]`, `<whisper>`, `<emphasis>`). Sub-second time-to-first-audio. $4.20 / 1M characters. Voice cloning available (deferred). Picked.
- **OpenAI TTS** — cheaper-ish, smaller voice library, no expressive tags. Fine, less interesting.
- **ElevenLabs** — best quality, ~40× more expensive per char, heavier SDK. Overkill for the "does this feel different?" question.
- **Cartesia Sonic** — comparable latency, smaller voice library, less mature for narration use cases.

The expressive tags are the deciding factor — they're the closest thing to "the narrator can sound theatrical" without going to ElevenLabs.

## Scope

No schema changes. One new route. One new lib module. One new component. Edits to `Chat.tsx` and `pricing.ts`.

```
routes:
  POST /api/tts                   → proxies one sentence to xAI TTS, streams audio bytes back

env:
  XAI_API_KEY                     NEW. server-side only, never exposed to client
  TTS_VOICE                       NEW. defaults to "eve"
```

### Files

```
src/lib/tts.ts                              NEW. xAI TTS REST client (server-side). Single function: streamSpeech(text, voice) → ReadableStream<Uint8Array>.
src/lib/sentence-splitter.ts                NEW. incremental sentence boundary detector for streaming text. Pure, no deps.
src/lib/pricing.ts                          + xAI TTS rate ($4.20/1M chars), + costForTts(chars) helper.
src/app/api/tts/route.ts                    NEW. POST { text, voice? } → audio/mpeg stream. Server-side, holds the API key.
src/components/Chat.tsx                     wires in useNarratorAudio hook, adds mute toggle to per-turn footer, shows speaker icon while playing.
src/components/useNarratorAudio.ts          NEW. client hook. Watches narrator turn text deltas, batches into sentences via the splitter, POSTs each to /api/tts, queues + plays the returned audio blobs sequentially. Owns the mute state (localStorage).
src/app/worlds/[worldId]/play/page.tsx      passes initial mute state from a cookie/localStorage to <Chat>.
```

No DB changes. No new tables. `turns.metadata` gains an optional `tts_chars` field on the narrator turn (whatever was sent to TTS), tracked by the same `onFinish` that already writes narrator usage. No migration needed because metadata is already JSON.

## Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Provider | **xAI Grok TTS** | Expressive tags, sub-second TTFA, cheap. The expressive tags are the only differentiator that matters for narration. |
| Transport | **REST server-streamed, one POST per sentence** | WebSocket would be lower-latency but requires a custom Node server (breaks Vercel-style deploys). Per-sentence REST is sprint-sized and the latency budget is fine: ~300–700ms per sentence is comfortably below the narrator's own token cadence. |
| Key handling | **Server-side proxy via `/api/tts`** | CLAUDE.md security rule: no API keys client-side. Non-negotiable. |
| Sentence detection | **Client-side regex on streaming text** | Server-side would mean buffering the narrator stream through our own route, adding complexity for no benefit. The browser already has the deltas. |
| Boundary rule | **`.` / `!` / `?` followed by whitespace or end, with a 200-char fallback flush** | Standard prose ends in punctuation; 200-char fallback catches "let me think…" style trailing fragments that never get a period. |
| Voice selection | **Single env var `TTS_VOICE` (defaults to `eve`)** | Per-world voice is a v0.5 concern (joins naturally with the characters table). Hardcoded is fine for testing whether TTS feels right at all. |
| Mute | **Per-browser, localStorage** | Not per-world; if you mute, you mute. No DB column. |
| Playback | **Native `<audio>` elements, queued, sequential** | MediaSource Extensions would give gapless playback but is finicky cross-browser. Brief gaps between sentences are acceptable and arguably aid comprehension. |
| Speech tags in narrator output | **Out of scope (stretch)** | Adding `[laugh]`/`<whisper>` to the narrator system prompt is a one-line edit but introduces a quality risk — narrator might overuse them. Try in v0.4.1 after we hear how the unadorned voices sound. |
| Replay / skip per-turn | **Out of scope** | Mute is the only control. Replay UI is its own design pass. |
| Cost ribbon | **TTS chars added to per-turn footer and session total** | Existing `pricing.ts` pattern. Per-turn footer becomes `narrator 1,240/380 · state 410/95 · tts 612 chars · ~$0.020`. |
| Failure mode | **TTS errors are silent** | If `/api/tts` fails (rate limit, network, key issue), the sentence just doesn't play. No retry, no error UI. The narrator turn already rendered as text; that's the source of truth. Console-log server-side for debugging. |

## Architecture sketch

```
┌─────────────────────────────────────────────┐
│ Browser                                     │
│                                             │
│ ┌─────────────────┐    ┌─────────────────┐  │
│ │ Chat.tsx        │───▶│ useNarratorAudio│  │
│ │ (useChat hook)  │    │  - splits sents │  │
│ │  text deltas    │    │  - POSTs each   │  │
│ └─────────────────┘    │  - queues audio │  │
│                        │  - plays seq    │  │
│                        └────────┬────────┘  │
└─────────────────────────────────┼───────────┘
                                  │ POST /api/tts
                                  │ { text, voice }
                                  ▼
                  ┌───────────────────────────┐
                  │ Next.js /api/tts (Node)   │
                  │  - reads XAI_API_KEY env  │
                  │  - calls xAI REST stream  │
                  │  - pipes raw audio bytes  │
                  └───────────┬───────────────┘
                              │ POST api.x.ai/v1/tts
                              │ Bearer XAI_API_KEY
                              ▼
                  ┌───────────────────────────┐
                  │ xAI Grok TTS              │
                  └───────────────────────────┘
```

The narrator turn is unchanged — text streams from `/api/chat` to `Chat.tsx` exactly as it does in 0.3. The audio path is a parallel, fire-and-forget loop attached to those same text deltas. If TTS fails entirely, play is unaffected.

## Explicit cuts (still deferred)

- `characters` table and per-character voices (now v0.5)
- WebSocket TTS transport
- Voice cloning
- Inline expressive tags in narrator system prompt
- Per-turn replay / skip / scrub UI
- Per-world voice selection in the create-world form
- Background music, ambient audio
- Player voice input (STT)
- Caching identical sentences (the cost won't matter at single-user scale; this is a Phase 2/3 problem)
- Mobile autoplay-policy handling beyond the standard "user-gesture-unlocks-audio" pattern

## Accepted tradeoffs

- **First-sentence latency is ~500–800ms after the first period appears.** Narrator may already be mid-second-sentence when audio starts. This is fine — the audio queue catches up.
- **Audio gaps between sentences are audible** (~50–150ms). MediaSource would close this; not worth the cross-browser pain in this sprint.
- **One voice for all worlds.** "Mevagissey 1897" and a sci-fi premise both narrate in `eve` until v0.5.
- **No graceful degradation if xAI is down.** The narrator still works (text streams normally); audio just silently doesn't play. Acceptable for a single-user dev tool.
- **No content filtering between narrator and TTS.** Whatever the narrator generates gets spoken. The narrator's existing content rules are the only line of defense.
- **`turns.metadata.tts_chars` may diverge from actual chars sent** if a turn is mid-stream when the user mutes. Cost shown is a best-effort estimate, not a billing audit.
- **No rate limiting on `/api/tts`.** Phase 5 concern. Single-user means a runaway loop is bounded by narrator output rate.

## Exit criteria

- Fresh `npm run dev` with `XAI_API_KEY` set → open a world → play a turn → narrator prose streams in AND is spoken sentence-by-sentence in the configured voice.
- Mute toggle silences current and future playback immediately. Refresh the page → mute state persists.
- Unset `XAI_API_KEY` → narrator still works, audio silently doesn't play, server logs the failure once per turn.
- Per-turn footer shows `tts NNN chars · ~$0.00X`. Session total in header includes TTS cost.
- 10-turn session: no console errors, no audio overlap, no stuck-on-playing state. Audio queue drains correctly even if the user types a new action mid-playback (queue is cancelled, new turn's audio takes over).
- `npm run lint` and `npm run type-check` pass.

When all six hold, tag `v0.4.0`. The next sprint (v0.5 candidate) takes the deferred `characters` table and pairs each character with a voice ID, so dialogue can render in distinct voices.
