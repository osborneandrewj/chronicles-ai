# v0.6.12 — Audio Latency & Feedback

**Target**: 2026-06-11
**Bar**: "The narrator starts speaking a beat or two after a turn lands, not after the whole turn has finished generating — there's a visible signal that audio is being prepared and how far along playback is — and it still reads like one continuous voice."
**Budget**: ~4–6 hours of focused work.

Patch on top of `v0.6.11` (audio cache & cost correctness). No schema changes, one optional env var. This milestone is about **time-to-first-audio (TTFA)**, a **playback/prep progress indicator**, and a residual **tone-consistency** spike. It does *not* touch the narrator, archivist, or world-state pipeline. It depends on v0.6.11 having plumbed the real DB turn id to the client and switched the cache to turn-based retention — that foundation is what makes the first-chunk-overlap design below cache-coherent.

> **Context — what prompted this.** An external review (Grok) flagged narrator audio as slow and recommended (1) switching from chunked TTS to a single full-text request, (2) replacing the playback hook with a `blob()`-then-`new Audio()` path, (3) a mount-time pre-warm, and (4) stability params for tone. We evaluated all four against the tree before planning. The short version: **(1) already shipped in v0.6.6** (we send one request per turn today), **(2) would regress TTFA** on Chrome/FF because it waits for the full MP3 before playing whereas we already stream progressively via MediaSource, **(3) as written is a billable xAI call on every page load** and must be done as a non-billable warm instead, and **(4) is a legitimate spike** but the worst seams it targets were already removed when we stopped per-sentence chunking. The genuine bottleneck the review missed: TTS does not start until the LLM stream *fully completes*.

## Why

1. **TTS doesn't start until generation finishes.** `useNarratorAudio.ts` gates on `if (effective.streaming) return;` and only then fires one `/api/tts` call for the full turn text. On a long turn the player stares at finished prose in silence for the whole xAI round-trip — which didn't even begin until the last token landed. This is the dominant TTFA cost and it's invisible to anyone reading only the playback code.
2. **No feedback while audio is being prepared.** The UI exposes only `idle | loading | speaking`. During the prep window the player gets no signal that audio is coming, how long, or how far playback has progressed — so the wait feels longer than it is and a slow turn is indistinguishable from a failure.
3. **The xAI route + connection pay a cold-start tax on the critical path.** The first request of a session (and after idle) eats serverless cold start + TLS + xAI TTFB, all *after* the player is already waiting, overlapped with nothing.
4. **Residual tone drift within a turn.** We send no pacing/stability parameters to xAI (`tts.ts` body is `text`/`voice_id`/`language`/`output_format` only). Cross-*sentence* seams are gone since v0.6.6, but a single long generation can still drift if xAI exposes a knob we're not using.

## Scope

No schema. One new optional env var. Touches the audio hook, the TTS lib, the TTS route, the splitter (revived from dead-code status), the chat page (warm + exposing streaming text/cursor and rendering the progress UI), and tests. The ordered-playback queue (`JobState.pending`/`nextSeq`/`playSeq`, `playNext`) is **unchanged** — it already supports multiple ordered clips; we only change what gets enqueued and when, and surface progress out of it.

```
env:
  TTS_PREWARM_VOICE            optional. Voice id used for the connection warm; defaults to DEFAULT_VOICE. Server-only.

routes:
  POST /api/tts?warm=1         existing route, new branch: warm the xAI connection without billable synthesis.
```

## Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Reject Grok's "single full-text request" rec | **Reject — already shipped** | v0.6.6 moved us to one request per turn. Re-recommending it is a no-op; Grok analyzed v0.5.1-era code. |
| Reject Grok's `blob()` + `new Audio()` rewrite | **Reject — it regresses TTFA** | `playProgressiveMediaResponse` already starts playback on first bytes via MediaSource. `await res.blob()` waits for the *entire* MP3. We'd trade away our one working latency optimization on Chrome/FF. (The blob path is already our Safari-only fallback.) |
| How to actually cut TTFA | **First-chunk overlap: synthesize paragraph 1 during generation; single request for the tail** | The floor for a pure single request is `full-generation-time + xAI-TTFB`. Overlapping the first paragraph drops perceived TTFA to `paragraph-1-time + xAI-TTFB`. The bulk of the turn is still one request, so prosody is preserved except for **one** seam at the para-1/tail boundary. (Confirmed with Andrew: the one-seam tradeoff is accepted.) |
| Why not full paragraph chunking (revive v0.5.1 wholesale)? | **No — one seam, not N** | v0.5.1 chunked every paragraph and was deliberately collapsed to single-request in v0.6.6 for prosody. We take the latency win only where it matters (the first chunk) and keep the tail whole. |
| First-chunk size | **First paragraph boundary at or after `minChars` (~280)** | A bare one-line opener would otherwise make chunk 1 trivially short and the seam land almost immediately with a huge tail. The floor makes chunk 1 a real prosodic unit. Tunable. |
| Cache coherence under overlap | **Replay reproduces the identical split; turn-based retention (from v0.6.11) holds both chunks** | The live turn now caches per-chunk (chunk1 hash + tail hash). Because the splitter is deterministic and replay runs it with the same `minChars`, replay requests the same two chunks → both hit v0.6.11's cache → free replay, no full-text/chunk divergence. v0.6.11's turn-based retention keeps *all* chunk entries for the newest 2 turn-ids, so a 2-chunk turn doesn't evict itself. |
| Progress: prep phase | **Indeterminate shimmer (upgrade to determinate iff xAI sends Content-Length)** | Chunked MP3 transfer likely has no Content-Length, so a true percentage isn't available pre-playback. Honest default: indeterminate "preparing" state, optionally a char-count ETA. Phase 0 instrumentation confirms whether Content-Length is present; if so, prep becomes a real byte-progress bar. |
| Progress: playback phase | **Determinate (currentTime/duration)** | Trivially accurate: `audio.currentTime/duration` for the media-element path, `elapsed/buffer.duration` for Web Audio. The hook exposes a single `progress` (0..1 or `null` for indeterminate). |
| Pre-warm | **Fire `POST /api/tts?warm=1` on player submit; warm the connection with NO billable synthesis** | Overlaps cold start + TLS + xAI handshake with generation. Grok's mount-time `"Testing audio."` is a paid xAI call per page load that counts against the daily cap — rejected. |
| What "warm" does if xAI has no warm/ping endpoint | **Open + immediately abort, or warm only our serverless route** | Pay the TCP/TLS/cold-start cost without generating audio. If xAI bills any request that reaches synthesis, fall back to warming only our route (still removes Next.js cold start from the critical path). Decided in Phase 0. |
| Stability / speed params (Grok #4) | **Spike first (Phase 0); add IFF supported** | We can't assert xAI's param surface from memory. If `speed`/`stability` exist, wire them with conservative defaults. If not, close the item as "not supported". |
| Cost accounting under two chunks | **Sum both chunk char counts into `tts.chars`, same as today** | `onTurnComplete` fires per non-cached chunk with `chunk.length`; two chunks sum to the full turn. Combined with v0.6.11's dollar split, the footer stays correct. No change needed here. |
| Instrument before optimizing | **Phase 0 ships TTFA breakdown logging first** | Grok guessed "/api/tts is the bottleneck". We measure: stream-end→request-sent, request-sent→first-byte, first-byte→first-audio, and whether xAI sends Content-Length. Optimize what the numbers say. |

## Architecture sketch

```
player submits action
   │
   ├──▶ POST /api/tts?warm=1   (non-billable; pays cold-start/TLS/xAI handshake in parallel)
   │
   └──▶ narrator stream starts ──▶ text grows token by token
                                      │
                  splitNewChunks(text, cursor, { minChars: 280 })
                                      │ first paragraph boundary at/after minChars?
                                      ▼  yes
                              chunk 1 ──▶ /api/tts ──▶ pending[0] ──▶ playNext()  ◀── FIRST AUDIO
                                      │                                   │
                                  (stream continues)            progress: prep→playback
                                      ▼ stream end (flush)
                              remainder ──▶ /api/tts ──▶ pending[1] ──▶ playNext()  (one seam here)

replay: cursor jumps to end, splitter reproduces [chunk1, tail] → both cache HITs (free, from v0.6.11)
```

## Explicit cuts

- **xAI streaming-input TTS** (feed text as it generates, zero seams + low latency) — out of scope; depends on xAI protocol support. Spike-note it in Phase 0; pursue later if it exists.
- **Per-character voices / speaker tags** — still deferred.
- **A user-visible "smooth vs. fast" toggle** — not worth the surface area; the `minChars` dial and the one-seam design are the compromise.
- **Reworking the three playback paths** (Web Audio / progressive MediaSource / Safari blob) — they work; consolidating them is not a latency win and is its own risk.

## Accepted tradeoffs

- **One prosodic seam per live turn.** At the para-1/tail boundary, xAI's two stateless generations may differ slightly in colour. Strictly better than v0.5.1 (a seam per paragraph) and the accepted price of overlap. Replays have no extra seam (they reproduce the same split).
- **Two cache entries per live turn instead of one.** Resolved by v0.6.11's turn-based retention (keeps all chunks for the newest 2 turn-ids) and deterministic replay (same split → cache hits). No extra synthesis at turn time.
- **First-chunk overlap reintroduces a cursor + splitter into the streaming path** — more moving parts than the current "wait, then one request". Mitigated: the queue infra it feeds is already present and tested; we revive a deleted call site.
- **Pre-warm adds one request per turn.** Non-billable (no synthesis) and short. If xAI bills any reaching request, downgrade to warming only our route.
- **Prep progress is indeterminate unless xAI sends Content-Length.** An honest shimmer, not a fake percentage. The first-chunk overlap keeps this phase short anyway.
- **`minChars` is a guess (~280).** One number, easy to tune after a listening test; not worth a config surface.

## Phase 0 findings (resolved during implementation)

Spike resolved against xAI's official TTS docs before wiring:

- **`speed` exists; stability does not.** xAI's only prosody knob is `speed` (rate multiplier, range `0.7`–`1.5`, default `1.0`). There is no `stability`/`temperature`/prosody parameter, so within-generation tone drift (#4) is **not tunable via the API**. We wired optional `speed` anyway (env `TTS_SPEED`, parsed by `resolveSpeed`; unset/invalid/out-of-range → no `speed` field → byte-identical request to v0.6.11), and recorded the stability finding. Resolves exit criterion 10.
- **No non-billable warm/ping endpoint.** Every request that reaches synthesis bills. So `?warm=1` does **not** call `/v1/tts`; `warmConnection()` issues a `HEAD https://api.x.ai` with a 2s timeout to pay DNS/TLS/cold-start only. Matches the planned fallback ("open + abort / warm only our route"). Guaranteed non-billable.
- **No documented `Content-Length` on the unary response.** Prep progress stays **indeterminate** (sweeping shimmer), as the plan's honest default; the determinate bar is playback-only (`currentTime/duration`).
- **`optimize_streaming_latency` exists** (trades chunk-boundary quality for faster TTFB) — noted as a future TTFA lever; out of scope here since first-chunk overlap is the primary mechanism.

Two implementation deltas from the file list above:
- **Splitter/hook tests live in `tests/`**, not co-located. Vitest's `include` is `tests/**/*.test.ts`, so the old `src/lib/sentence-splitter.test.ts` never ran; it was migrated to `tests/sentence-splitter.test.ts` (+ `tests/tts-request-body.test.ts`, `tests/tts-route-warm.test.ts`). No jsdom harness exists, so the hook's overlap/progress logic is covered through the (pure, tested) splitter plus manual dev verification, as the plan permitted.
- **`minChars` mode takes no soft-cap forced cut.** Chunk 1 is emitted only at a real paragraph boundary at/after `minChars`; the soft-cap mid-paragraph cut would decide differently between a partial stream and the full-text replay, breaking cache coherence. A turn that is one giant paragraph simply doesn't overlap (single tail chunk) — deterministic and equal to v0.6.11 behaviour.

## Exit criteria

1. `npm run lint` and `npm run type-check` pass.
2. `npm test` passes, including new splitter cases (`minChars` floor, first-chunk-during-stream, tail-on-flush, replay-reproduces-split) and any hook tests.
3. `package.json` **and** `package-lock.json` read `0.6.12` on the release branch (both version fields in the lock), committed on the release branch — not post-merge. Restart `npm run dev` and confirm the `/` header shows `v0.6.12`.
4. **TTFA instrumentation (Phase 0) is visible**: a dev-only log prints stream-end→request, request→first-byte, first-byte→first-audio, and whether xAI sent Content-Length. Optimization targets match the numbers.
5. In `npm run dev`, submit an action that produces a ≥3-paragraph turn: audio begins **while the narrator is still streaming** (chunk 1 fires before `streaming` flips false), observable as a `/api/tts` request that starts before the stream completes.
6. Exactly **one** `?warm=1` request fires per submitted action, returns 204, and produces **no** billable xAI synthesis (confirm against the cost footer not moving for the warm).
7. The UI shows an audio-prep indicator during the wait and a determinate playback bar once speaking; the bar reaches 100% exactly as playback ends, then clears.
8. Subjective listening test: the para-1/tail seam sits at a paragraph boundary and reads acceptably; the tail is a single continuous read.
9. Replay one of the last two turns → splitter reproduces the same two chunks, both return `X-TTS-Cache: HIT`, no xAI call, footer voice cost unchanged (proves overlap + v0.6.11 cache compose correctly).
10. Stability-param spike resolved: either conservative `speed`/`stability` defaults are sent and a turn sounds no worse, or the doc/Decisions record "xAI does not expose it" and `tts.ts` is unchanged on that axis.
11. Deploy → production smoke test: Andrew + Joe each submit one live turn, confirm audio starts mid-stream with a visible progress indicator, no 5xx, daily cost cap still gates.
12. Tag `v0.6.12` on `main`.

When all twelve hold, v0.6.12 ships. The next audio milestone is the real endgame — xAI streaming-input TTS (if it exists) to remove the last seam and the per-chunk request entirely — sketched only as a Phase 0 spike note here.
