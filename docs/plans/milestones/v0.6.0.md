# v0.6.0 Milestone

**Target**: 2026-05-25 (shipping date)
**Bar**: "Two characters and a place exist as rows, not as fragments of a JSON blob. When I play a turn, the archivist updates the right row; when I open the inspector, I see who the narrator thinks is in the scene; when I give an NPC a gift today and meet them three sessions later, they remember it."
**Budget**: ~12–16 hours of focused work (original spec); ~3 additional hours for the v0.6.0 rebase/audit pass on 2026-05-25.

> **Origin note.** This doc was authored as a v0.5 milestone on 2026-05-24 (commit `95dbe85` on `chore/remove-basic-auth`) and code-completed at that time, but never merged. v0.5.0 shipped without it (see [`docs/plans/milestones/v0.5.0.md`](v0.5.0.md)), v0.5.1 shipped audio chunking, v0.5.2 shipped docs canon, and the parked typed-state work was rescued and shipped as v0.6.0 on 2026-05-25. Three additional commits were applied to the parked branch before the rebase to address audit findings: `IF NOT EXISTS` on v5 DDL (idempotency), a malformed-`turn_states.state_json` test (production-likely failure mode), and `[t:N]` turn-id provenance on `memorable_facts` writes (cheap groundwork for v0.7 embeddings).

v0.4 added voice to the narrator. The deploy sprint put it in front of one other human. v0.5.0 swapped Basic-auth for Cloudflare Access; v0.5.1 chunked TTS by paragraph; v0.5.2 canonised the docs. v0.6.0 is the structural shift the long-term docs have been pointing at since the start: **stop pretending the world is a 3-field JSON blob**. The `turn_states` row (`time` / `location` / `identity`) was always a placeholder for typed entity tables. This release replaces it.

Vector memory and per-character voices remain deferred — see [Explicit cuts](#explicit-cuts) and the v0.7 note at the end. The "characters remember things" requirement is tested with the cheapest possible mechanism (free-text `memorable_facts` per character, maintained by the archivist, with deterministic turn-id provenance). If that proves insufficient through real play, v0.7 promotes it to embeddings — and the `[t:N]` suffix added in this release makes that migration mechanical.

## Why

Three concrete problems with the current model:

1. **The state blob caps the world at one player, one location, one moment.** NPCs the narrator invents have no row to attach to. The next time you meet them, they're whoever the narrator hallucinates from recent turns.
2. **"Authoritative state" is a single line of identity prose.** It can't say "Joe is in the tavern, you're at the harbour" because there's no second row to hold Joe's location.
3. **There is nowhere to put a memorable fact.** Gifts, promises, debts, scars — anything the narrator should remember about a specific entity — currently lives only in the prose of recent turns, which scroll out of context after ~20 turns.

The fix is the schema from `docs/specs/database-design.md` §3, narrowed to the three entities a single-player solo novel actually needs: characters, scenes, places. Relationships, threads, wiki pages, and timeline events stay deferred until the data they need is being written.

### Schema

```sql
-- v5 migration

CREATE TABLE characters (
  id                INTEGER PRIMARY KEY AUTOINCREMENT,
  world_id          INTEGER NOT NULL REFERENCES worlds(id) ON DELETE CASCADE,
  name              TEXT    NOT NULL,
  description       TEXT,
  is_player         INTEGER NOT NULL DEFAULT 0,         -- SQLite boolean
  current_place_id  INTEGER REFERENCES places(id) ON DELETE SET NULL,
  memorable_facts   TEXT,                                -- newline-delimited free text, archivist-maintained
  voice_id          TEXT,                                -- column added; not wired until v0.7
  status            TEXT    NOT NULL DEFAULT 'active' CHECK (status IN ('active','inactive','dead')),
  traits_json       TEXT,                                -- optional structured traits; archivist may write partial JSON
  created_at        TEXT    NOT NULL DEFAULT (datetime('now')),
  updated_at        TEXT    NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX characters_world_id ON characters(world_id);
CREATE UNIQUE INDEX characters_world_name ON characters(world_id, lower(name));

CREATE TABLE places (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  world_id        INTEGER NOT NULL REFERENCES worlds(id) ON DELETE CASCADE,
  name            TEXT    NOT NULL,
  description     TEXT,
  kind            TEXT,                                  -- free-form: "harbour", "tavern", "ship's deck"
  created_at      TEXT    NOT NULL DEFAULT (datetime('now')),
  updated_at      TEXT    NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX places_world_id ON places(world_id);
CREATE UNIQUE INDEX places_world_name ON places(world_id, lower(name));

CREATE TABLE scenes (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  world_id        INTEGER NOT NULL REFERENCES worlds(id) ON DELETE CASCADE,
  place_id        INTEGER REFERENCES places(id) ON DELETE SET NULL,
  title           TEXT    NOT NULL,
  summary         TEXT,                                  -- one or two sentences, archivist-written at scene close
  scene_number    INTEGER NOT NULL,
  status          TEXT    NOT NULL DEFAULT 'active' CHECK (status IN ('active','completed')),
  opened_at_turn  INTEGER REFERENCES turns(id) ON DELETE SET NULL,
  closed_at_turn  INTEGER REFERENCES turns(id) ON DELETE SET NULL,
  created_at      TEXT    NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX scenes_world_id ON scenes(world_id);
CREATE UNIQUE INDEX scenes_world_number ON scenes(world_id, scene_number);

ALTER TABLE turns ADD COLUMN scene_id INTEGER REFERENCES scenes(id) ON DELETE SET NULL;
CREATE INDEX turns_scene_id ON turns(scene_id);

ALTER TABLE worlds ADD COLUMN current_time TEXT;        -- promoted from turn_states.time
ALTER TABLE worlds ADD COLUMN current_scene_id INTEGER REFERENCES scenes(id) ON DELETE SET NULL;

-- Backfill: each world gets one player character, one opening place, scene 1
-- populated from worlds.initial_state_json (the old time/location/identity blob).
-- All existing turns are reassigned to scene 1.

DROP TABLE turn_states;
```

The migration follows the same rebuild-style pattern as v4 for tables that need new FK columns. `runMigrations` already disables FKs around the run.

### Archivist patch shape

The archivist returns a Zod-validated object, not free text. The apply step is pure DML — no model in the loop.

```ts
type ArchivistPatch = {
  current_time?: string                                   // optional update
  scene?:
    | { keep_open: true }
    | { close: { summary: string } }
    | { open: { title: string; place_name: string } }    // place upserted by name
  places?: Array<{ name: string; description?: string; kind?: string }>
  characters?: Array<{
    name: string
    is_player?: boolean
    description?: string
    current_place_name?: string                           // FK by name, resolved server-side
    memorable_facts_append?: string                       // single line, appended with newline
    status?: 'active' | 'inactive' | 'dead'
  }>
}
```

The patch is small on purpose. The archivist sees the prior structured state plus the latest two turns and decides what changed. It cannot retract `memorable_facts` (append-only within v0.6.0; cleanup is a v0.7 problem if it becomes one). It cannot delete rows. It cannot rewrite a scene's history — only close it and open a new one.

## Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Storage | **SQLite, no change** | Vector retrieval is v0.7's problem. Doing the Postgres migration alongside the schema rewrite triples the sprint and gives one painful migration window instead of two. |
| Entity scope | **characters, places, scenes only** | Three rows cover the "two characters and a place" bar. Relationships, threads, wiki pages stay deferred until something writes to them. Each new table without a writer is a maintenance tax. |
| Memory model | **`characters.memorable_facts` as newline-delimited text, each line suffixed with `[t:N]`** | Cheapest possible test of "do typed notes solve the gift-remembering case?" before reaching for embeddings. Append-only inside the sprint. If it works, v0.7 keeps it. If it doesn't, v0.7 promotes to chunked + embedded rows — and the `[t:N]` suffix gives v0.7 deterministic source-turn provenance for each fact, making the embedding pass mechanical. |
| Archivist transport | **Single `generateObject` call with a Zod patch schema** | Same shape as today's `extractState`. The model decides what changed; deterministic apply code writes to the tables. Keeps creative output (narrator) and factual extraction (archivist) split, per CLAUDE.md. |
| Patch granularity | **Append-only within a turn** | No row deletes, no `memorable_facts` retraction, no scene retcons. The archivist can make mistakes; we cap the blast radius to "extra row" rather than "missing history". Repair tool ships in v0.7 alongside the CRUD UI. |
| Scene transitions | **Archivist proposes; route applies** | Archivist returns `scene: { close, open }` only when the latest turns clearly establish a place/time change. Default is `keep_open`. Avoids "every turn is a new scene" inflation. |
| Place identity | **Lookup by lowercased name within a world** | No place-name UUIDs in the patch. Cheaper for the model to reason about, and the `UNIQUE (world_id, lower(name))` index enforces dedup at the DB layer. Cost: name collisions across genres in one world (two "Black Anvils") are impossible until v0.7 introduces aliases. |
| Player character | **Created at world creation, `is_player = 1`** | The current `identity` blob is just a player-character row in disguise. New world form gains "Your character" and "Opening location" fields. Old worlds get backfilled from `initial_state_json`. |
| Voice column | **Added but unused** | Per [[deploy-sprint-v0-4-1]] + earlier v0.4 plan, voices were the original deferred item. The column lands so v0.7 can populate it without another migration; the wiring (speaker tags, voice routing) is v0.7's job. |
| Authoritative state block | **Built from typed rows, formatted by `world-state.ts`** | New block shape: current scene + place + present characters (each with their memorable facts) + world time. The narrator never sees a JSON blob — only formatted markdown, same convention as the current `formatStateBlock`. |
| Inspector | **Read-only drawer, refreshes on turn completion** | Andrew + Joe need to see what the archivist is writing to trust the pipeline. CRUD is a v0.7 question; v0.6.0 gives debug visibility without committing to an editing UX. |
| Inspector trigger | **Toggle button in `<Chat>` header** | Same row as the audio toggle. Persisted-open state in localStorage; default closed so the play surface stays focused. |
| Backfill | **Synthetic single character / single place / scene 1 per existing world** | Andrew has at least one Mevagissey world running. The migration must not break it. Source the synthetic row contents from the most recent `turn_states.state_json` (or from `worlds.initial_state_json` if there are no turn states). |
| Delete safety | **All FKs are `ON DELETE SET NULL` for back-references, `CASCADE` only from `worlds`** | A wrong-place row should not nuke the scene that referenced it. World deletion still cleans up everything (matches v4 behavior). |
| Drop `turn_states` | **Yes, in the same migration** | Two state systems coexisting (per the rejected scoping option) is a long-term tax. The new typed rows are now the source of truth; the old blob has no readers after this sprint. |
| Replay cost tracking | **Additive update to the replayed turn's `tts.chars`, no separate field** | A replay re-sends every sentence to xAI — that's real money that should show in the cost ribbon. Two design choices: (1) target by `turnId` instead of "latest assistant turn" so replaying an old turn updates the right row; (2) sum into the same `tts.chars` field instead of adding a `tts.replay_chars` sibling. Sum keeps the per-turn footer + session total honest with one number. If the user replays a 1,200-char turn three times, the footer shows `tts 4,800 chars · ~$0.02`. Cost: you can't see "original vs. replayed" without a separate field, which is fine — the question we care about is "what did this turn actually cost?" |

The narrator prompt is unchanged in shape — same system prompt, same EPHEMERAL_CACHE on the trailing messages — but the state block it receives is now built from rows instead of `formatStateBlock(parseState(...))`. The cache key changes once on deploy; that's a one-time cost.

## Explicit cuts

Deferred, not deleted:

- **Vector embeddings of turns / scenes.** v0.7.
- **Semantic retrieval of past turns into the narrator prompt.** v0.7.
- **`relationships` table.** Wait until two NPCs talk to each other often enough that "who likes whom" stops being legible from `memorable_facts`.
- **`timeline_events` table.** Same logic — defer until something queries it.
- **`story_threads` table.** v0.7 or whenever a quest log UI matters.
- **`wiki_pages` table.** Phase 2 work; the archivist isn't writing structured wiki entries yet.
- **NPC turns / Actor agent.** Narrator still produces all dialogue. Characters are nouns in v0.6.0, not speakers.
- **Per-character TTS voices wired up.** Column exists; nothing reads it. v0.7 adds speaker tags + routing.
- **CRUD / editing UI for entities.** Inspector is read-only. Andrew or Joe can edit rows by hand via Railway shell if needed; v0.7 adds an editor.
- **Multi-place scenes / split-party scenes.** One scene = one place. NPCs not in the current place are excluded from the state block.
- **Patch validation against canon contradictions.** The archivist can update a character's `current_place_name` to somewhere that contradicts the narrator's last sentence. No linter catches this; we eyeball it via the inspector.
- **Archivist retry / repair on extraction failure.** If `generateObject` throws, the turn still saves; the next turn's archivist call sees stale state. Best-effort, same posture as the current extractor.
- **Per-user world scoping.** Joe and Andrew still share the world list.

## Implementation deviations from the spec above

Two intentional changes made during build, kept here so the next person grepping for them isn't surprised.

- **Column is `worlds.world_time`, not `worlds.current_time`.** `CURRENT_TIME` is a SQLite reserved keyword that silently shadows a bare-identifier column reference in SELECTs, returning the system clock instead of the column value. The JSON patch field returned by the archivist is still `current_time` — only the SQL column name changed.
- **Scene action is a discriminated union, not the tagged-key shape.** Schema:
  ```ts
  scene?: { action: 'keep_open' }
        | { action: 'close', summary: string }
        | { action: 'open', title: string, place_name: string }
  ```
  `generateObject` produces cleaner output against a discriminator than against a union-by-which-key-is-present. Also: `open` auto-closes the prior active scene (no summary) instead of requiring a separate `close` action in the same patch. The doc's intent — "close + open in one patch" — is satisfied; the shape is just simpler.

## Accepted tradeoffs

- **`memorable_facts` is append-only and unstructured.** If the archivist appends "received a silver locket from the player" three times across three sessions, the entry just repeats. The `[t:N]` provenance suffix means duplicates are at least identifiable from the column alone; v0.7's promotion to chunked + dedupable storage is the structured cleanup path. For v0.6.0, repetition is acceptable: the narrator can re-see the same fact safely.
- **Place identity by name is fragile.** The archivist returning "the harbour" vs "Mevagissey Harbour" creates two rows. The unique index dedupes exact case-insensitive matches; near-matches silently fork. We watch for this in the inspector and tune the prompt rather than building canonical resolution in v0.6.0.
- **Scene boundaries are squishy.** "When does a scene end?" is a judgment call the archivist gets wrong. Default-to-`keep_open` biases towards under-segmentation; if the inspector shows 20 turns in scene 1 with five obvious location changes, the prompt needs work, not a code fix.
- **The archivist call gets more expensive.** Larger schema, more context (it sees the typed rows it's updating). Budget: ~30% more Haiku tokens per turn. Well inside the `DAILY_TOKEN_LIMIT = 200000` cap; expect ~$0.005/turn instead of ~$0.003/turn for the extractor.
- **Backfill is best-effort.** Existing worlds get one synthetic character / place / scene seeded from the legacy blob. Names will read as "Player" / "Opening Scene" unless Andrew edits them post-migration. The Mevagissey world will need a one-time tidy-up in Railway shell; new worlds get clean rows.
- **No transaction across narrator turn + archivist patch.** Narrator turn writes commit on stream finish; archivist patch commits ~1-3s later when extraction returns. A page reload in the gap sees the narrator turn without the state update. Refresh in 5 seconds and it's there. Same posture as today's `extractState`.
- **The inspector adds an extra HTTP round-trip per turn.** Refetches on `onFinish`. ~50ms. Not user-visible; not optimizing further.
- **Voice column is dead weight for one sprint.** Acceptable to avoid a second migration in v0.7.
- **Drop of `turn_states` is irreversible without restoring from a Railway volume snapshot.** Take a manual snapshot before deploying the v5 migration. The migration itself runs inside a transaction (per `runMigrations`), so a failed migration leaves the DB at v4 — but a successful migration on bad backfill data is harder to undo. Snapshot first.

## Exit criteria

1. Fresh `npm run dev` → create a new world (with player name + opening place fields) → play a turn → `SELECT * FROM characters` shows the player row, `SELECT * FROM places` shows the opening place, `SELECT * FROM scenes` shows scene 1 with `status = 'active'`.
2. Play a turn that introduces a new NPC by name → the archivist creates a `characters` row for them; their name is visible in the inspector without a page reload.
3. Play a turn where the player gives an NPC something → that NPC's `memorable_facts` gains a line referencing the gift. Three turns later (or after a refresh), the narrator's state block still includes that line; the narrator references the gift when asked.
4. Migrate the existing Mevagissey world (in dev, against a copied SQLite file) → backfilled rows are sensible, no foreign-key violations, all existing turns reassigned to a scene with the right `world_id`. `pragma foreign_key_check` returns no rows.
5. Inspector toggle in `<Chat>` header opens a right-side drawer; drawer shows characters / places / scenes; refreshes on turn completion; closes without unmounting `<Chat>`.
6. Deploy to Railway → Andrew + Joe both play one turn each on the live URL → no 5xx, daily token cap still gates correctly, inspector loads on both browsers.
7. Replay an old narrator turn → that turn's `tts NNN chars · ~$0.0XX` segment in the per-turn footer grows by the replay's char count, and the session total in the header grows by the matching cost. Replay an older turn (not the most recent) → the correct turn's footer updates, not the latest one.
8. `npm run lint` and `npm run type-check` pass. `npm test` passes (test suite for `archivist.ts` patch application + a migration test that backfills a synthetic v4 DB).
9. Tag `v0.6.0` on `main`.

When all nine hold, v0.6.0 ships. The next sprint (v0.7 candidate) addresses episodic memory: chunk + embed turns or `memorable_facts` lines (the `[t:N]` suffix makes the source turn for each fact recoverable deterministically), retrieve by similarity at the top of `/api/chat`, and wire `characters.voice_id` so dialogue routes to distinct TTS voices. v0.7 is also when the Postgres migration becomes worth doing — pgvector + the existing schema is a one-time pain that unlocks both retrieval and concurrent multiplayer.
