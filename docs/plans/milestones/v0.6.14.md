# v0.6.14 — Story Conductor v1 (escalation / the director) — STUB

> **Stub.** Scoped after `v0.6.13` ships and the latent-hook bridge proves out in play. Numbers, schema, and exit criteria below are provisional.

**Target**: TBD (after v0.6.13).
**Bar**: "When I ignore a thread, the world doesn't just sit there — pressure builds. A lead comes looking for me, stakes rise, or something off-screen moves. I'm pulled back toward a story without being railroaded."
**Budget**: TBD (~12-16 hours, provisional).

## Why this is its own milestone

v0.6.13 delivered the **seed** half of "pull toward stories": the world is populated and carries latent hooks (continuation + seed) connecting the ambient world to the dossier. But hooks in v0.6.13 are **passive** — they surface where the player happens to be and wait to be pulled. Nothing escalates if the player drifts.

The **escalate** half is an *active pacing decision*: detect drift, choose to push a hook toward the player, raise stakes, move the world off-screen. Per the Decide → Render → Record split established in `v0.6.13`, this is **Decide** work that must **not** live in the narrator (a writer, with an 8K budget that can't hold drift history or off-screen world state). It belongs to the **Story Conductor** — an agent that already exists in the design as a stub returning `{ action: 'proceed' }`. This milestone wakes it for the first time.

## Provisional scope

- **Drift detection** — a deterministic signal for "the player has not advanced any active thread in N turns / is far from every tagged place." Cheap, code-side, feeds the Conductor.
- **Conductor decision** — given drift + dossier + current occupancy/hooks, the Conductor decides one of: `proceed`, `surface` (promote a latent hook's visibility / strength), `approach` (send a relevant NPC or lead toward the player), `escalate` (raise a thread's stakes / advance a clock), or `interlude` (off-screen world beat). Small structured output; Haiku-tier.
- **First off-screen advancement** — a single, bounded "the world moved while you were elsewhere" beat tied to an active thread, surfaced through the dossier/narrator on the next relevant turn.
- **Boundary with the NPC agent** — **NPC Agent = tactical** (how tracked NPCs behave this turn); **Conductor = strategic** (should the story escalate / transition / introduce pressure). Resolve the overlap explicitly here.

## Likely cuts (provisional)

- Full event-clock / story-clock system — only the minimum needed to escalate one thread.
- Multi-thread orchestration / arc scheduling.
- Relationship graph, semantic memory — still deferred.
- Background cron simulation — escalation still runs in the chat path, not on a timer.

## Open questions for scoping

1. What is the drift signal exactly — turns-since-thread-advance, distance-to-tagged-place, or both?
2. How aggressive should escalation be by default, and is it player-tunable (difficulty / "railroad-ness")?
3. Does `approach` reuse the v0.6.13 promotion path (a latent occupant becomes the lead who finds the player), or move an existing tracked NPC?
4. How does the Conductor avoid thrashing — escalating, then the player engages, then it escalates something else next turn?

## Exit criteria (provisional)

1. Lint / type-check / tests pass, including Conductor decision coverage and drift-detection unit tests.
2. Conductor is no longer a stub: a real decision is produced and acted on each turn (default still `proceed` when no drift).
3. Manual: ignore an active thread for N turns and observe an `approach`/`surface`/`escalate` response that pulls toward it without hard railroading.
4. Manual: verify a bounded off-screen advancement surfaces coherently.
5. `package.json` / `package-lock.json` read the milestone version on the release branch; header confirms after dev-server restart.
6. Deploy to Railway, smoke one turn, tag on `main`.
