# v0.6.11 — Audio Cache & Cost Correctness

**Target**: 2026-06-04
**Bar**: "Replaying either of the last two narrator turns costs $0 in voice, and the per-turn footer shows text and voice spend as separate dollar figures that update the instant you (re)generate audio — no page reload."
**Budget**: ~3–4 hours of focused work.

Patch on top of `v0.6.10`. No schema changes, no new env vars. This milestone fixes a correctness-and-money bug: narrator audio caching and TTS cost tracking silently do nothing for freshly-streamed turns. It is the foundation the latency work in `v0.6.12` builds on — that milestone's first-chunk-overlap design is only cache-coherent because this one plumbs the real DB turn id to the client and switches the cache to turn-based retention.

> **Context.** While planning the audio-latency work, we traced three separate symptoms Andrew reported — "caching is broken", "regenerated audio cost isn't reflected", "can't see text vs voice cost" — to a single root cause described in *Why* #1. The dollar machinery already exists (`costForTts`, `$4.20/M chars`, folded into `TurnCost.total`); it just never receives a valid turn id for the turns that matter.

## Why

1. **A freshly-streamed turn has no DB turn id on the client.** The AI SDK assigns the streamed assistant message a generated string id (`msg-…`); the real DB id (`narratorTurn.id`) is created server-side in `chat/route.ts` `onFinish` but never sent back — the route ends with a bare `result.toUIMessageStreamResponse()`. History-loaded turns get `id: String(t.id)` (numeric); live turns don't. Three downstream failures cascade from this:
2. **Audio cache never populates for live turns.** The hook sends `numericTurnId(turnId)` → `undefined` for a non-numeric id → `/api/tts` `parseCacheRef` returns `null` → it neither stores nor looks up audio. Caching only ever works after a page reload, so replaying the turn you just heard re-bills xAI.
3. **TTS cost never reaches the header for a live or regenerated turn.** `reportTtsChars` does `Number(turnIdStr)` → `NaN` → bails before calling `/api/tts/record`, so `metadata.tts.chars` stays 0 and the footer's voice cost is invisible until (and unless) a reload happens. Separately, the footer renders voice as raw `tts NNN chars` rather than a dollar figure, so text vs voice spend can't be compared at a glance.

## Scope

No schema, no migration, no env vars. One server-side change (expose the DB turn id), client-side id resolution + footer rendering, and a cache retention-policy change from entry-count to turn-count. The TTS request/response shape, the audio queue, and the pricing model are all unchanged.

  POST /api/chat              + attach { dbTurnId } to the streamed assistant message via messageMetadata

## Decisions

| Decision | Choice | Rationale |
|---|---|---|
| How to get the DB turn id to the client | **`messageMetadata` on the streamed message** | The metadata channel is already wired (`MessageMetadata` carries `createdAt` today), so this is additive and keeps the AI-SDK message id stable. Annotate rather than rewrite. |
| Alternative considered: rewrite the message id to the DB id post-stream | **Rejected as primary; kept as fallback** | Reassigning a live message's id mid-session can remount the row and fight `useChat`'s id stability. Use only if the messageMetadata finish-ordering proves unworkable. |
| Ordering risk: id must exist when finish-metadata serializes | **Capture `narratorTurn.id` into a closure var at the top of `onFinish`; `messageMetadata` finish-part reads it** | `insertTurn` already runs first thing in `onFinish`. The verify step (exit criterion 4) confirms the streamed message actually arrives with `dbTurnId`; if the SDK serializes finish metadata before `onFinish` runs, fall back to the post-stream id-reconcile. |
| Retire `buildCostMap`'s end-align fallback? | **Keep it** | It's cheap defense for retry transients and for any turn that arrives without `dbTurnId`. Belt and suspenders. |
| Cache retention policy | **Turn-based: newest 2 distinct turn_ids per world (was: newest 3 entries)** | "Last 2 turns" is the user's mental model and is robust to v0.6.12's multi-chunk-per-turn caching (a 2-chunk turn won't evict its own first chunk). Entry-count retention silently breaks once a turn is more than one entry. |
| Why 2 turns, not 3 | **2 is the stated requirement; audio blobs are ≤8 MB each** | Bounded storage, matches the ask. Trivially bumpable via one constant. |
| Voice cost display | **Dollars (`voice ~$Y`), char count in the title attr** | Matches every other footer segment (all dollars) and makes text-vs-voice directly comparable. Raw chars stay available on hover for debugging. |
| Charge manual regenerate? | **Record + refetch only on cache MISS** | The hook already calls `onTurnComplete` only when `X-TTS-Cache !== HIT`, so a cached replay is correctly free and a miss (evicted turn, new voice) records and surfaces. The only gap is the footer not refreshing after a non-stream replay — fixed by refetching usage after `reportTtsChars`. |

## Architecture sketch

```
stream finishes ──▶ onFinish: insertTurn → narratorTurn.id ──┐
                                                              ├─▶ messageMetadata({finish}) = { dbTurnId }
                    toUIMessageStreamResponse ◀───────────────┘
                          │
client receives assistant message with metadata.dbTurnId
                          │
        ┌─────────────────┼────────────────────────┐
        ▼                 ▼                         ▼
 narratableTurn.id   reportTtsChars(dbId)     replay(dbId, text)
   → hook sends        → /api/tts/record        → /api/tts with valid
     valid turnId        + refetchUsage            turnId → CACHE HIT
        │                 (footer updates)          (free)
        ▼
 /api/tts caches audio keyed (world, turn, model, voice, textHash)
        │
 prune keeps audio for the newest 2 turn_ids
```

## Explicit cuts

- **First-chunk overlap, pre-warm, stability params, progress bar** — all deferred to v0.6.12 (audio latency & feedback). This milestone is cache + cost only.
- **Caching audio for arbitrary back-catalogue turns** — only the newest 2 turns are retained; scrubbing far back and replaying re-synthesizes. Out of scope.
- **A session-total voice/text breakdown in the header** — the per-turn footer split lands here; a global split (if wanted) is a separate small follow-up.

## Accepted tradeoffs

- **Replaying older-than-2 turns still costs.** Turn-based retention caps at 2; an intentional replay of an evicted turn re-bills xAI (and now correctly shows the cost). Acceptable — the ask was specifically the last two.
- **One added field on the wire.** `dbTurnId` rides on every streamed assistant message. Negligible size; the channel already exists.
- **Footer reads dollars, not chars, for voice.** Anyone who relied on the raw char count now hovers for it. Worth it for at-a-glance comparability.
- **The finish-metadata ordering is SDK-version-sensitive.** If a future AI SDK bump changes when finish metadata serializes relative to `onFinish`, the fallback id-reconcile path is the safety net; exit criterion 4 will catch a regression.

## Exit criteria

1. `npm run lint` and `npm run type-check` pass.
2. `npm test` passes, including new retention tests (2 turns kept, 3rd evicted) and id-resolution tests.
3. `package.json` **and** `package-lock.json` read `0.6.11` on the release branch (both version fields in the lock), committed on the release branch — not post-merge. Restart `npm run dev` and confirm the `/` header shows `v0.6.11`.
4. In `npm run dev`, stream a fresh turn: the assistant message arrives with `metadata.dbTurnId` set to the DB turn id (verify in React devtools or a temporary log), and the `/api/tts` request for that turn carries a numeric `turnId`.
5. Immediately after a fresh turn streams, hit Replay → the `/api/tts` response shows `X-TTS-Cache: HIT`, no xAI synthesis occurs, and the footer's voice cost does **not** increase.
6. After a fresh turn's live synthesis, the footer shows `text ~$X · voice ~$Y` with `voice > 0` **without a page reload**.
7. Replay a turn whose audio was evicted (older than the last 2) → footer voice $ increases and the header total updates live, no reload.
8. Replay each of the last 2 turns → both `X-TTS-Cache: HIT` (free); replay the 3rd-from-last → `MISS`, proving 2-turn retention.
9. Deploy → production smoke test: Andrew + Joe each replay their last turn → no cost bump, footer shows the text/voice split.
10. Tag `v0.6.11` on `main`.

When all ten hold, v0.6.11 ships. The latency and feedback work — first-chunk overlap, non-billable pre-warm, the prep/playback progress bar, and the xAI stability spike — is specified in `docs/plans/milestones/v0.6.12.md` and builds directly on the turn-id foundation laid here.
