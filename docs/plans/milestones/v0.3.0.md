# v0.3 Milestone

**Target**: 2026-06-22 (~2 weeks from 0.2.0)
**Bar**: "I can create a new world from the UI, play it, switch to a different one, and the two don't bleed into each other."
**Budget**: ~8–10 hours of focused work.

0.1 proved the streaming loop. 0.2 closed the per-turn quality gaps (classifier, retry, cost). 0.3 finally opens the schema: a `worlds` table, a `world_id` foreign key on turns and turn-states, world CRUD pages, and per-world routing. Premise and initial state move off code constants and onto the world row, so authoring a new world stops requiring an edit to `src/lib/prompt.ts`.

This is deliberately the *minimum* schema opening. Characters and scenes stay deferred to v0.4. We're not porting to Postgres or Drizzle yet — that re-platform lands when embeddings (Phase 2/3) actually need it.

## Scope

One new table. Two new FK columns. Three new routes.

```
schema:
  worlds(id, name, premise, initial_state_json, created_at)
  turns         + world_id NOT NULL REFERENCES worlds(id)
  turn_states   + world_id NOT NULL REFERENCES worlds(id)

routes:
  GET  /                          → list worlds (replaces the single-chat home)
  GET  /worlds/new                → create-world form (Server Component + Server Action)
  GET  /worlds/[worldId]/play     → existing chat UI, scoped to one world
  POST /api/chat?worldId=…        → existing streaming route, scoped to one world
  GET  /api/usage?worldId=…       → existing usage route, scoped to one world
```

## Decisions

| Decision | Choice | Rationale |
|---|---|---|
| New tables | **`worlds` only** | Characters and scenes defer to v0.4. One structural change at a time. |
| `world_id` on existing rows | **NOT NULL with backfill in migration** | A nullable FK invites the bug we're trying to prevent (cross-world leakage). Backfill all extant rows into a seeded "Mevagissey 1897" world before adding the constraint. |
| DB stack | **SQLite + raw better-sqlite3** | Postgres/Drizzle is a Phase 2/3 cost, not a v0.3 cost. We're not re-platforming in the same sprint we open the schema. |
| World identification in routes | **Path segment `/worlds/[worldId]/play`** | Matches `docs/specs/api-design.md`. The API route reads it from query string because the AI SDK transport posts to a single URL. |
| Initial state authoring | **Form fields: name, premise, location, identity, time (defaults to "Day 1, morning")** | Premise and location are required because the narrator drifts hard without them. Identity defaults to "Travel-worn newcomer — name not yet established." so the player can earn it in play. |
| Backwards compatibility | **None for `PREMISE` / `INITIAL_STATE` consts** | Delete them. The world row is the only source. No re-export shim. |
| World list ordering | **`ORDER BY created_at DESC`** | Newest first; the user lands on the most recent world they were tinkering with. |
| Delete-world | **Out of scope** | Single-user dev tool; manual SQLite delete is fine. Adds cascade complexity that doesn't earn its keep. |
| Cost ribbon | **Scoped per-world** | Cross-world totals would be misleading once two worlds exist. Header shows current world's totals only. |
| Slash commands & `/pause` | **Unchanged** | Meta-command handler stays world-agnostic — `/pause` still bypasses DB by design. |

## Migration plan (v4)

Single transaction. Order matters:

1. `CREATE TABLE worlds (...)`.
2. Insert one row: name = "Mevagissey 1897", premise = current `PREMISE` const verbatim, initial_state_json = serialized current `INITIAL_STATE` const. Capture its `id` as `DEFAULT_WORLD_ID`.
3. `ALTER TABLE turns ADD COLUMN world_id INTEGER REFERENCES worlds(id)`.
4. `UPDATE turns SET world_id = DEFAULT_WORLD_ID`.
5. Recreate `turns` with `world_id NOT NULL` (SQLite can't add NOT NULL to an existing column — standard create-new + copy + drop + rename dance).
6. Same recreate dance on `turn_states`.
7. Indexes: `CREATE INDEX turns_world_id_id ON turns(world_id, id)`, `CREATE INDEX turn_states_world_id_turn_id ON turn_states(world_id, turn_id)`.

The migration is destructive of the existing single-table assumption but non-destructive of existing turn data — a fresh `npm run dev` against an existing `chronicles.sqlite` lands the user in the same Mevagissey session they had at 0.2.

## Explicit cuts (still deferred)

- `characters` table, player character editing, name profiles, naming policies
- `scenes` table, tactical_state metadata, scene transitions
- Archivist, embeddings, retrieved memory
- World deletion, world editing after creation, premise diffing
- Auth, multi-user isolation, rate limiting per user
- Playwright smoke test, regression fixture from `docs/reference/example-chat-narrative.md`
- Mobile polish beyond what already works
- Server Actions infra (the create-world form uses a Server Action, but no broader Action framework is built)

## Accepted tradeoffs

- **Worlds are not editable after creation.** Premise typo → create a new world. Premise authoring polish is its own sprint.
- **Anyone with the URL plays.** Still no auth; two browsers on `/worlds/3/play` share the same chat. That's a Phase 5 problem.
- **Initial state schema is still rigid (time / location / identity).** When characters and scenes land in v0.4, identity moves onto the player character row and the world row keeps only setting-level fields.
- **No migration rollback.** SQLite migrations here are forward-only; the backfill is the only safety net.
- **State extractor needs the world's premise injected per call.** The existing `EXTRACTOR_SYSTEM` template captures `PREMISE` at module load — now it accepts the premise as a parameter on every call. Small breaking change to the function signature, fully internal.

## Exit criteria

- Fresh clone, fresh DB → `npm run dev` → `/` shows "No worlds yet" with a Create World button.
- Create a world named "Test Harbour" with a one-paragraph premise → redirected to `/worlds/[id]/play` → play 3 turns. Premise from the form is visible in narrator behavior (it's grounding the prose).
- Back to `/` → both "Test Harbour" and any pre-existing "Mevagissey 1897" appear in the list, newest first.
- Play 2 turns in Test Harbour, then 2 turns in Mevagissey 1897 → each world's turn count, cost total, and authoritative state stay independent. No leakage in either direction.
- Existing `/pause` and slash-command palette still work in both worlds. Cost ribbon shows per-world totals. Retry button still recovers a failed stream.
- `npm run lint` and `npm run type-check` pass.

When all four hold, tag `v0.3.0`. The next sprint (v0.4 candidate) adds the `characters` table — moving identity off the world row and onto a per-world player character — and a minimal player-character creation flow inside the create-world form.
