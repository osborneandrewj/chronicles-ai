# v0.6.3 Milestone

**Target**: TBD (after v0.6.2 ships)
**Bar**: "Narrator audio plays on Safari/WebKit, including iOS Safari — both on first-turn-after-load and after the tab is backgrounded. No silent failures."
**Budget**: small — single file, three targeted changes, plus an optional diagnostic panel.

v0.6.2 ([`docs/plans/milestones/v0.6.2.md`](v0.6.2.md)) shipped the narrative-style/observation-memory work. v0.6.3 is a focused patch release for one bug: narrator audio is silent on Safari/WebKit. The mobile/desktop UI pass and the replay-cost-footer bug originally bundled here were split out so the audio fix could ship without waiting on the mobile design call; they now live in v0.6.5 ([`docs/plans/milestones/v0.6.5.md`](v0.6.5.md)).

## Why

**Narrator audio is silent on Safari/WebKit.** Smoke-tested 2026-05-25 on the v0.6.1 Railway deploy: tapping the audio toggle to "Audio on", sending a turn, and tapping any prior turn's Replay button all produce **total silence — no `speaking` indicator, no sound** on iOS Safari. Later testing reports the same broken narration path on desktop Safari, while Chrome still works. Codex confirmed via Railway HTTP logs that `/api/tts` returns 200s on mobile (audio bytes are arriving), so the failure is downstream of the network.

Root cause candidate: the `AudioContext` is still in `suspended` state when `source.start()` fires asynchronously after the TTS fetch resolves — Safari silently drops playback in that case. The v0.5.1 Web Audio rewrite (commit `749e3f8`) was meant to fix exactly this by replacing `HTMLAudioElement.play()` with `AudioContext` + `AudioBufferSourceNode`, but it was never verified on real Safari/WebKit devices. That change also moved MP3 playback onto `ctx.decodeAudioData(...)`; Safari's Web Audio MP3 decoder can be pickier than Chrome's. During implementation, the resume/longer-prime/context-lifetime changes were not enough on Safari, so the shipped fix keeps Web Audio for Chrome and uses one reused native `HTMLAudioElement` for Safari MP3 blob playback. It also stops paragraph-based TTS dispatch: each narrator turn now sends one full-response `/api/tts` request after streaming finishes.

## Scope

One workstream. No new tables, no new agents, no new env vars.

                                        ~368, `if (ctx && ctx.state !== "closed") void ctx.close()`).
                                        AudioContext close is unrecoverable; once closed a new context

                                        `source.start()` when `ctx.state === "suspended"`. Current

                                        calls `source.start()` — on Safari/WebKit, start() against a
                                        suspended context can be silently dropped. Make playNext's

                                        least ~50ms — instead of the current 1-sample buffer

## Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Safari/WebKit audio fix shape | **Keep Chrome on Web Audio; use a reused native `HTMLAudioElement` for Safari/WebKit** | The original three Web Audio hardening changes are still useful, but Safari remained silent after them. The practical fix is to avoid Safari's Web Audio MP3 decode/playback path entirely while preserving Chrome's working path. Reusing one media element keeps the playback object tied to the submit/unmute/replay gesture instead of creating a new element per async TTS chunk. |
| Safari audio diagnostic panel | **Optional, opt-in via `?debug=audio`** | Wanted by Andrew during the 2026-05-25 session because we didn't have the iPhone-Mac dev-tools chain set up. Behind a query param so default UX is unchanged. Records context state, resume result, decode result, and play-start/play-end so we can distinguish decode failures from suspended-context drops. Cut the file after the fix is verified unless it's still useful. |
| Split from mobile UI + replay-cost work | **Audio fix ships alone as v0.6.3; mobile + replay-cost later moves to v0.6.5** | Audio is the most urgent (Joe's smoke test is gated on it) and is a single-file change. Bundling it with a UI design pass would delay the fix. Splitting also keeps the diff readable. |

## Open questions

- **Does fix #2 alone resolve it, or do we need #1+#3 too?** Plan is to ship all three since they're all cheap; if the diagnostic panel shows fix #2 is sufficient, fix #1 (don't-close-ctx) is still worth keeping on its own merits (closing then reopening a ctx on hook remount is a latent bug). Fix #3 (longer prime buffer) is harmless either way.
- **Is any remaining Safari failure a decode failure rather than an unlock failure?** If `/api/tts` returns bytes and `ctx.resume()` resolves to `running`, but no buffer reaches `play-start`, instrument `ctx.decodeAudioData(...)` and compare with a native `<audio>` blob playback fallback. Safari may accept the MP3 through a media element while rejecting or mishandling the same bytes through Web Audio.

## Explicit cuts

- **Mobile + desktop UI pass** → moved to v0.6.5.
- **Replay-cost-footer bug** → moved to v0.6.5.
- **No new agents, no new tables, no new env vars.** v0.7 territory.

## Accepted tradeoffs

- **No automated tests for the fix.** Web Audio context state transitions don't replay reliably in jsdom; verification is manual on real Safari/WebKit devices.
- **The diagnostic panel may stick around past v0.6.3 if it proves useful for the v0.6.5 replay-cost-footer debugging.** Acceptable — the `?debug=audio` gate means default UX is unchanged.

## Exit criteria

1. `npm run lint` and `npm run type-check` pass.
2. `npm test` passes (existing tests; no new automated coverage in this milestone).
3. `package.json` reads `0.6.3` on the release branch (bumped on the branch itself per [[feedback_release_version_bump]]).
4. Manual (Safari audio — fresh session): on iPhone Safari and desktop Safari → open a world → ensure header shows "Audio on" → send a turn → narrator response streams in **with audible playback**; speaking indicator (amber dot) fires. Then tap Replay on any prior turn → audible playback. **No silent failures.** This is the smoke test that uncovered the bug; it's the canonical pass condition.
5. Manual (Safari audio — backgrounded session): on iPhone Safari and desktop Safari with audio working, switch away for 30s, return → next send still produces audible audio (verifies fix #2 handles the suspended-ctx race after backgrounding, not just on first load).
6. Manual (Chrome regression): on desktop Chrome, all v0.6.2 audio behaviour still works — toggle, streaming playback, replay.
7. Deploy to Railway → Andrew + Joe each play one turn on iPhone Safari. Audio audible. No 5xx.
8. Tag `v0.6.3` on `main`.
