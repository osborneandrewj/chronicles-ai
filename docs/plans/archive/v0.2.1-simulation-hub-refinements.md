# Plan: v0.2.1 — Simulation-Hub Refinements

Status: **ready to build** · branch `onion-arch-refactor` · target version **v0.2.1**
Builds on the v0.2.0 simulation-hub (`docs/plans/simulation-hub-implementation-plan.md`, Phases A–D, behind `SIM_HUB`).

Four post-playtest refinements from the "Iteration Cradle / Underground Bunker" run. All gated by the existing `SIM_HUB` flag (these only touch the hub path).

## Locked decisions (owner, this session)

1. **Terminology — keep `world` / `subworld` in code.** No rename of the engine aggregate or the layer enum (`world_layer` values stay `hub|subworld|standalone`). Fix the *user-facing copy* only: the UI says **"Simulations"** and **"Hub"**, never "subworld". Low risk, no migration.
2. **Completed simulations — read-only transcript.** From the hub, a list of past simulations; opening one shows its narrative read-only (no composer). The character died there; it's a record to revisit, not re-enterable.
3. **Hub name — from the Meta-Story Bible.** The bible generates a short in-fiction `institutionName` (e.g. "The Vesper Program"); the hub world is named that. The archetype's literal label ("Underground Bunker") stays internal (ensemble/era dressing only) and never reaches the player.
4. **Narrator opens the hub turn** on the awakening (and any later hub arrival).

No schema migration is required — `world_layer` / `parent_world_id` / `meta_story_json` already exist; `institutionName` rides inside `meta_story_json`.

## Item 2 — Home list redesign + in-hub simulation archive + read-only review

**The problem:** post-awakening the home list shows both the hub and the simulation; clicking the simulation bounces to the hub (`resolveActiveWorldId`), orphaning the simulation's narrative.

### 2a. Session-aware home list
`src/lib/worlds.ts` + Mongo `mapWorldSummary` + `domain/entities/world.ts`:
- Add `world_layer` to `WorldSummary` and the `listWorlds` SELECT (SQLite) + the Mongo summary mapper. (Column already exists; just project it.)

`src/app/page.tsx` — replace the current "hide unawoken hub" filter with, per world (using `sessions.byWorld(id)`):
- **standalone** → always show.
- **subworld** → show at top level ONLY while it is the *active, concealed* simulation (`session.status==='in_subworld' && session.has_awoken===0 && session.subworld_world_id===id`). Once awoken, hide it (it moves into the hub archive).
- **hub** → show ONLY once `has_awoken===1` (revealed). Hidden while concealed (today's behavior).

Net effect: one top-level entry per playthrough — the active simulation before the reveal, the hub after.

### 2b. "Past Simulations" panel in the hub
- `WorldRepository.simulationsForHub(hubId)` (NEW port method) → `WorldSummary[]` of worlds with `parent_world_id = hubId`, newest-first. SQLite + Mongo adapters.
- `src/app/worlds/[worldId]/play/page.tsx`: when rendering a **hub** (`world.world_layer==='hub'`), fetch `simulationsForHub(worldId)` and pass it to a NEW `HubSimulationsMenu` client component (a collapsible panel / drawer in the hub view, beside the inspector). Each entry links to the review route (2c). Empty state: "No completed simulations yet."

### 2c. Read-only review route
- NEW `src/app/worlds/[worldId]/review/page.tsx` (server): loads the world + its turns (reuse `latestTurns` / the play page's loader) and renders the transcript read-only. **Does NOT run** the `resolveActiveWorldId` redirect (this is a deliberate review). Renders only the simulation's own turns — no hub/parent data — so concealment is unaffected.
- `src/components/Chat.tsx`: add an optional `readOnly` prop that hides the composer, slash menu, and the `router.refresh()` completion effect; keeps message rendering + audio playback. The review page renders `<Chat readOnly … />`.
- Guard: 404 if the target world is not a `subworld` (review is sim-only).

**Done when:** post-awakening the home list shows only the hub; the hub view lists past simulations; clicking one opens its full narrative read-only with no composer; the live play flow is unchanged.

## Item 3 — Hub named from the Meta-Story Bible

- `domain/entities/meta-story.ts`: add `institutionName: string` (a short proper name for the institution, distinct from the longer `institution` description).
- `prompts/meta-story.md`: instruct an `institutionName` — "a short, evocative proper name for the institution/program (2–4 words), e.g. a designation or codename the staff use; must NOT name the archetype (no 'bunker'/'ship'/'lab')."
- `grok-meta-story-generator.ts` Zod schema + `stub-meta-story-generator.ts`: add `institutionName` (stub derives a deterministic neutral name, e.g. `"${codeword} Program"`).
- `app/worlds/new/actions.ts` `createAdventureAction`: **reorder** so the bible is generated *before* the hub world is created, then name the hub from it:
  1. pick archetype → 2. generate bible (pass `hubName: archetype.name` as context) → 3. `createBoundedWorld({ name: bible.institutionName ?? generateCodename(seed), premise: hubPremise, … })` → 4. `setLayer(hub,'hub')` + `setMetaStory(hub, bible)` → 5. session → 6. `enterSubworld`.
  - Bible generation stays best-effort: on failure, hub name falls back to a `generateCodename(seed)` (opaque) — never the spoiler archetype label.
- Existing hubs (e.g. "Underground Bunker") are not retro-renamed; optional one-off repair noted in the PR, not required.

**Done when:** a freshly created hub's player-facing name is the bible's institution name (or an opaque codename on bible failure), never the archetype label; the archetype label appears nowhere player-visible.

## Item 4 — Narrator takes the first hub turn on awakening

`ReturnToHub` already opens an "Awakening" scene but leaves it empty. The reveal should land with prose.

- `src/app/worlds/[worldId]/play/page.tsx`, in the existing awakening branch (right after `returnToHub`, before the redirect): generate the hub's opening narrator turn, then redirect. Reuse `generateOpeningTurn(openingTurnDeps, hubWorldId, awakeningPremise)`.
  - `awakeningPremise` = the hub premise + an awakening frame ("the protagonist has just surfaced from the simulation into the [sim room]; the friendly crew are here; this is the moment the truth begins to show"), kept genre-neutral. The narrator's existing state block (player in the sim room, present crew, the `REALITY` cue) grounds it.
  - Idempotent + one-time: only generate if the hub's active scene has no assistant turn yet (guards against a double render / re-awakening).
- Trade-off noted: this runs an LLM call during the awakening transition (a server-component render). It fires exactly once per awakening (guarded by `has_awoken` + empty-scene check), so the cost is bounded; if it proves too slow we move it behind the chat route. For later hub *returns* (the future simulation loop), the same empty-active-scene check seeds a fresh opening.

**Done when:** awakening lands the player in the hub's simulation room with a narrator opening turn already on the page (not an empty scene).

## Test strategy
- **Pure/services:** unchanged signatures; add a unit test for the home-list visibility rule if it's extracted to a pure helper (recommended: `domain/services/world-visibility.ts` taking `(world_layer, session)` → boolean, so it's testable without the page).
- **Repos:** `simulationsForHub` under SQLite (`npm test`) + Mongo (`npm run test:mongo`); `WorldSummary.world_layer` round-trips in both.
- **Meta-story:** stub returns a non-empty `institutionName`; `createAdventureAction` reorder doesn't break creation (existing create tests).
- **Review route / readOnly Chat:** lightweight — assert the review page renders turns and omits the composer (component-level or a DOM assertion).

## Exit criteria
1. `npm run lint`, `npm run type-check`, `npm run depcruise` pass.
2. `npm test` + `npm run test:mongo` green (incl. new visibility + `simulationsForHub` tests).
3. `SIM_HUB=1` manual: create → play a simulation → die → **arrive in the hub with a narrator opening turn**, hub named by the bible (no archetype label) → home list shows only the hub → hub view lists the past simulation → opening it shows the read-only transcript.
4. `SIM_HUB=0`: open-world + bounded paths unchanged.
5. Version bumped to **v0.2.1** on the release branch (workspace + root + contracts + lockfile together), dev server restarted, header verified.
