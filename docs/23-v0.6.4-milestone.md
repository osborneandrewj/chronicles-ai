# v0.6.4 Milestone

**Target**: TBD
**Bar**: "When the narrator refers to the same person by a long form once and a short form twice, the archivist canonizes them as a single `characters` row — not two — and goal/attitude state attaches to the canonical row consistently."
**Budget**: TBD — size after a small spike on the alias approach.

v0.6.1 added NPC `active_goal` and `current_attitude`; play-testing on 2026-05-25 surfaced a pre-existing archivist limitation that v0.6.1 made more visible (since duplicate rows now carry visible state, not just description). v0.6.4 closes the canonicalization gap. v0.6.3 (mobile UI + replay cost footer) is unrelated and stays on its own track.

## Why

1. **The archivist matches characters by `lower(name) = lower(?)` only.** When the narrator's prose alternates between "Professor William Henry Armitage" and "Armitage" across turns, the archivist creates two `characters` rows for the same person. Same shape for places (`"Mevagissey Harbour"` vs `"the harbour"`).
2. **v0.6.1 made the bug feel sharper.** Pre-v0.6.1, duplicate character rows were cosmetic — just an inspector-side annoyance. With `active_goal` + `current_attitude` on `characters`, each duplicate carries its own goal state. Andrew's Arkham world has two Armitage rows (ids 6 and 8) carrying compatible-but-distinct goals: "understand what Osborne knows" vs "provide directions to the Gilman place".
3. **The archivist prompt's mitigation is too weak.** Current text: *"Use the exact name the narration uses; do not paraphrase."* Narrators don't follow this — they naturally vary diction across turns, especially with titled characters (Professor / Dr / Father / Inquisitor / Sergeant).

## Scope

One archivist-side change + one schema option, plus tests. No new tables in the simplest version; an alias table is option B below.

```
A. Soft-match in the archivist (preferred minimal scope)
  src/lib/archivist.ts          EDIT. Extend findCharacterByNameStmt to fall back on a
                                token-overlap match when the exact lower-case lookup misses.
                                Heuristic: if all tokens of the candidate name (≥4 chars) are
                                present in an existing character's name (or vice versa),
                                treat it as the same character. Bias toward false-negative
                                (create a new row) when ambiguous.
  prompts/archivist-system.md   EDIT. Strengthen the "use the same name consistently" rule:
                                pick the shortest stable surface form on first canonization
                                and stick to it; explicitly list "Professor X" / "X" as a
                                pair that should collapse to "X".
  tests/archivist.test.ts       EDIT. Cover: full-then-short, short-then-full, title-prefix
                                drift, ambiguous case ("Tom" vs "Tom Penhaligon" — same row;
                                "Tom" vs "Tom Wright" — different rows).

B. Alias table (if A proves too fuzzy in real play)
  src/lib/migrations.ts         EDIT. v7 migration: character_aliases(character_id, alias).
  src/lib/archivist.ts          EDIT. Lookup tries aliases before insert; archivist can
                                emit an alias_for field on the patch.
  src/lib/archivist.ts          EDIT. CharacterPatchSchema gains an optional `aliases`
                                string array; apply writes to character_aliases.
```

A is the smaller hammer. B is the cleaner long-term shape. Start with A; promote to B if A misfires.

## Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Soft-match vs alias table | **A first, B if A misses** | A is one file + a prompt tweak; B is a migration + schema + a new patch field. Start with the cheapest version that could plausibly work; promote only on evidence. |
| Match threshold | **All tokens (≥4 chars) of the shorter name must appear in the longer** | "Tom" → matches "Tom Penhaligon"; "Tom" → does NOT collapse with "Tom Wright" because both are short and tokens differ. Tunable. |
| Migration of existing duplicate rows | **Manual one-shots per world via SQL** | The bug is rare today; a generic dedupe migration risks merging genuinely-distinct characters who share substrings. Document the manual procedure in this doc once we ship A. |
| Player-row exemption | **Yes — only match NPC-to-NPC** | The player is created at world-creation with a known name. The archivist should never collapse an NPC into the player row even if the NPC's name happens to overlap. |
| Logging | **console.warn on every soft-match hit, with both names** | Lets Andrew + Joe see when A is firing and validate it isn't aliasing too aggressively. |

## Explicit cuts

- **Place canonicalization** — same bug, same fix shape. Defer to a separate sub-task or land alongside A if the diff stays small. Out of scope here unless trivial.
- **Retroactive merge of existing duplicate rows** — handled by manual SQL per world for now. A bulk migration is more risk than reward.
- **Alias UI in the inspector** — v0.7+. No way to edit aliases from the inspector in v0.6.4.

## Accepted tradeoffs

- **Soft-match can occasionally merge two distinct characters.** Worst-case: "Tom Penhaligon" and "Tom Penhall" collapse to one row. Mitigation: tokens-≥4 + every-token-must-match heuristic plus the player-row exemption; console.warn so Andrew can spot it during play. If misfires cluster, promote to option B (explicit aliases) — that path was always going to be necessary if the player wanted to author aliases themselves.
- **Manual dedupe of existing rows is per-world toil.** Acceptable because the bug has been surfaced in exactly one row today (Arkham Miskatonic). v0.6.4's ship-condition is "no new duplicate rows", not "all historical duplicates are cleaned".
- **Archivist prompt drift back to long forms.** Strengthening the prompt rule helps but won't bind perfectly. The soft-match in code is the actual guarantee.

## Exit criteria

1. `npm run lint` and `npm run type-check` pass.
2. `npm test` passes; new archivist tests cover the four match cases (full→short, short→full, title-prefix drift, ambiguous-distinct).
3. `package.json` reads `0.6.4` on the release branch.
4. Manual: in a fresh world, the narrator introduces a character as "Professor X" turn 1 and refers to them as "X" turn 2. Inspector shows ONE characters row.
5. Manual: in a fresh world, "Tom Penhaligon" turn 1 and "Tom Wright" turn 2 produce TWO characters rows (no false merge).
6. Manual: run the documented one-shot SQL on Andrew's Arkham world; the duplicate Armitage rows collapse to one row carrying the chosen goal/attitude.
7. Deploy → Railway smoke: one turn end-to-end, no 5xx.
8. Tag `v0.6.4` on `main`.

When all hold, v0.6.4 ships.

## Open questions

- **What does the soft-match do when the narrator introduces a brand-new character whose name overlaps an existing one** (e.g. existing "Tom"; narrator now writes "Tom Wright" intending a new character)? Current heuristic says "collapse if all tokens overlap" — so a NEW "Tom Wright" would NOT collapse with existing "Tom" because "Wright" is new. But existing "Tom Penhaligon" + narration of "Tom" WOULD collapse. That's the intended asymmetry; worth confirming in test.
- **Should the inspector visually mark a row as "canonicalized from N variants"?** Useful for diagnosis. Probably v0.7.
- **Does the same shape work for places?** Almost certainly yes ("the harbour" / "Mevagissey Harbour"). The decision is whether to land both in v0.6.4 or split places into v0.6.5.
