# v0.5.1 — Audio Chunking

**Target**: patch release after `v0.5.0` ships and the v0.4.2 audio-hook hotfix lands.
**Bar**: "A turn reads aloud with the prosody of a paragraph, not the prosody of nine unrelated sentences glued together."
**Budget**: ~2–3 hours.

## Why

Per-sentence TTS gives xAI no context across sentence boundaries. Voice colour, pacing, and intonation reset every full-stop, so a multi-sentence beat reads as nine voiceover takes spliced together instead of one continuous read. The cheapest fix is to feed xAI larger units — paragraphs — so prosody decisions span the natural unit of narrator prose.

## Scope

One splitter change, one buffer-cap bump in the TTS route, no schema or DB changes, no new env vars.

```
src/lib/sentence-splitter.ts            EDIT. Rename to splitNewChunks; boundary becomes paragraph
                                        (blank line) with sentence-level fallback when a paragraph
                                        exceeds the soft cap. Export stays compatible.
src/components/useNarratorAudio.ts      EDIT. Call site rename only; semantics unchanged. Variable
                                        rename: sentence → chunk, fetchSentence → fetchChunk.
src/app/api/tts/route.ts                EDIT. Bump MAX_TEXT_CHARS from 2000 → 4000 to fit a
                                        comfortable paragraph; still well under xAI's per-call cap.
src/lib/sentence-splitter.test.ts       EDIT. New cases: paragraph boundary, soft-cap fallback,
                                        flush on stream end mid-paragraph.
```

## Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Chunk unit | **Paragraph (`\n{2,}`)** | Matches the natural prosodic unit. The narrator already emits blank lines between paragraphs; we have a free boundary signal. |
| Soft cap | **600 chars per chunk** | A run-on paragraph would otherwise delay first audio by 4–6s. At 600 chars, fall back to the existing sentence boundary inside the chunk and ship what we have. |
| Hard cap | **MAX_TEXT_CHARS = 4000 at the route** | Above the soft cap, below xAI's limit. Defence-in-depth against a pathological input the splitter mishandles. |
| Streaming behaviour | **Wait for paragraph boundary OR soft cap OR stream end** | First-audio latency goes from ~one sentence to ~one paragraph. Accepted: see tradeoffs. |
| Cost accounting | **Unchanged — sum chunk char counts into `tts.chars`** | Char count per turn is the same regardless of chunking; the per-turn footer keeps reading right. |
| Replay path | **Reuses the same splitter** | Replay sees the full text at once and chunks it identically. No separate code path. |
| Voice consistency across chunks | **Out of scope** | xAI still has no cross-call memory; chunk boundaries will still have a seam. Per-character voices + a streaming TTS protocol are v0.6. v0.5.1 reduces seam frequency, doesn't eliminate it. |
| Rename `sentence-splitter` file | **No, keep the path** | Avoid touching the broader import surface for a behaviour-only change. Export the new function alongside, deprecate the old name in a comment, drop in v0.6. |

## Architecture sketch

```
streaming text ──▶ splitNewChunks(text, cursor, { flush })
                    │   boundary = /\n{2,}/   (paragraph)
                    │   fallback = sentence boundary when chunk > 600 chars
                    │   flush    = remaining tail on stream end
                    ▼
                   chunks: string[]   ─┐
                                       ├─▶ /api/tts (one call per chunk)
                                       │     audio blob
                                       │     queued in JobState.pending[seq]
                                       ▼
                                    playNext() (unchanged)
```

`useNarratorAudio.ts` keeps its ordered queue + sequential playback unchanged. The only thing that changes is what arrives at each `seq`.

## Accepted tradeoffs

- **First-audio latency rises.** Today: ~one short sentence (~50–120 chars) before audio starts. After: ~one paragraph (~300–500 chars typical). Estimated additional wait: 1–2s on a fast turn, ~3s on a slow one. The prosody win is worth it; if it isn't in real use, the soft cap is the dial.
- **Seams still exist between paragraphs.** xAI is stateless across calls. A paragraph-to-paragraph voice shift is still possible, just rarer than sentence-to-sentence. Real fix is streaming-protocol TTS in v0.6.
- **The soft-cap fallback can split mid-paragraph.** A 1,200-char paragraph becomes two ~600-char chunks; the seam in the middle will sound like the old behaviour. Acceptable for the long-tail case; the median paragraph is well under 600.
- **MAX_TEXT_CHARS bump means an upstream bug could send a larger request.** The route still rejects above 4000; the splitter still caps each chunk at 600. Bound stays tight in practice.
- **No A/B toggle.** A user-visible "old chunking" switch isn't worth the surface area for a fix we believe in. If the change is wrong, revert the splitter.

## Exit criteria

1. `npm test` passes including the new splitter cases (paragraph boundary, soft-cap fallback, mid-paragraph flush).
2. `npm run lint` and `npm run type-check` pass.
3. In `npm run dev`, play a turn whose narrator output contains ≥3 paragraphs; observe in the network panel exactly one `/api/tts` request per paragraph (plus any soft-cap subdivisions), not one per sentence.
4. Subjective listening test on the same turn: voice/prosody is consistent within a paragraph; seams sit at paragraph boundaries rather than mid-beat.
5. Replay an old turn → same chunk pattern as the original stream; per-turn footer `tts NNN chars` still grows by the replay's char count (no regression vs. the v0.5 replay-cost behaviour).
6. Deploy → Andrew + Joe each listen to one live turn → confirm the audio reads as paragraphs, no 5xx, daily cap still gates.
7. Tag `v0.5.1` on `main`.

When all seven hold, v0.5.1 ships. The remaining audio work — per-character voices via `characters.voice_id`, speaker tags, and a streaming TTS protocol that eliminates inter-chunk seams entirely — stays queued for v0.6.
